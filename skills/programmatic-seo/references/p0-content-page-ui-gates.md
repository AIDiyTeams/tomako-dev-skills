# P0 内容页 UI 门槛

## 使用时机

规划、生成、重写、诊断或验收 Blog / Template / MCP 等内容型 SEO 页面时，必须读取本文件。它补充 `p0-content-page-structure-gates.md` 和 `content-page-implementation.md`，专门约束内容页如何呈现得像 Tomako 产品页面，而不是裸 Markdown 文档。

## 核心判断

内容页不是低配 Tool 页，也不是普通博客皮肤。内容页的职责是用可抓取正文承接搜索需求，再通过产品化 UI 把用户自然带到 Tomako 工具、模板或下一步。

必须区分：

- **MDX / Markdown**：负责内容、结构、frontmatter、链接、表格、图片引用。
- **React Content Shell**：负责 Hero、阅读布局、侧栏 CTA、相关工具、目录、FAQ 样式、视觉模块和品牌质感。
- **Skill / 模型**：负责关键词判断、文案、信息结构、图片 prompt 和验收。

禁止把 MDX 当作页面 UI 方案本身。只把 MDX 正文渲染成 `max-w-3xl` 的普通文章，视为内容页 UI 不合格。

## 与工具页 UI Gate 的关系

`p0-ui-gates.md` 中以下规则对内容页也成立：

- 先确定信息和文案，再设计 UI。
- 不要为了版式保留无价值模块。
- 不要左右两边重复讲同一件事。
- 不要用阅读时间、关键词标签、metadata 卡替代用户价值。
- 不要滥用 icon。
- Hero 不能只是标题加右侧无意义信息卡。
- 页面必须体现产品价值、解决用户疑问，并提供下一步。

以下规则是 Tool 页专属，不能硬套内容页：

- 表单最多 5 个外露字段。
- 左表单、右侧工作台图、下方结果区。
- 点击生成后滚到结果区。
- 结果区状态机、复制、下载、继续修改。
- 生成按钮全宽。
- 表单控件、下拉、tab、chip 的交互规则。

内容页如果没有交互 widget，就不要伪造工具工作台、假结果区或输入输出卡。

## 内容页默认页面层

内容页必须具备可复用的产品化外壳，而不是每篇 MDX 复制一套 HTML。

默认结构：

1. 顶部导航：返回栏目或上游 hub。
2. 产品化 Hero：H1、intro、分类、更新时间、主关键词或页面任务。
3. 价值摘要：读者在本页能得到什么，不能只展示 reading time。
4. 主正文区：可读行长、清晰 H2/H3、表格和列表有足够层级。
5. 侧栏或中段 CTA：相关 Tool / Template /下一步，不只放结尾。
6. 主题信息：关键词、cluster、父工具或相关主题。
7. 结尾 CTA：把用户自然带到工具、模板、相关指南或 hub。

## Hero

内容页 Hero 的目标是让用户确认“这篇页面回答我搜的问题”，并感受到这是 Tomako 的产品体系，而不是随机博客。

必须做到：

- H1 直接匹配搜索任务或核心问题。
- intro 说明读完能得到什么、适用边界和下一步价值。
- Hero 不要用右侧阅读时间、更新时间、分类卡作为主要视觉。
- 如果有 Hero 图，图片服务页面主题和用户价值，不替代 H1/intro。
- 法律、隐私、合规、竞品等高风险内容要在 Hero 或前 20% 内容中体现边界。

不要把“你会得到什么”做成大标题右侧 metadata 卡。它可以在 Hero 下方、侧栏、摘要卡或首段后出现。

## 阅读布局

内容页的正文要像专业产品资料，而不是裸文档。

必须做到：

- 正文有可读宽度，但不要把整个页面所有内容都压成窄列。
- 宽屏可以使用正文 + 侧栏；移动端侧栏下移。
- H2/H3、表格、引用、图片、提示块和 CTA 有统一样式。
- 长表格必须可横向滚动且边界清楚。
- 站内链接样式清晰，但不要全篇到处都是强下划线造成噪音。
- FAQ、清单、步骤和对比内容尽量用可扫描样式，而不是连续大段文字。

## 内容页组件

内容页可以逐步补充共享 MDX 组件，但不要每篇手写一次。

优先沉淀这些组件：

- `ContentPageShell`：内容页外壳。
- `ContentExploreCta`：相关工具/模板/下一步 CTA。
- `Callout`：重要提示、边界、警告。
- `StepList`：步骤或流程。
- `ComparisonTable`：对比和判断标准。
- `FaqBlock`：FAQ。
- `RelatedContent`：相关内容和内链。
- `ContentImage`：稳定尺寸和统一样式的正文图。

如果组件尚未存在，可以先用 Markdown 实现内容，但不能因此忽略页面外壳和 CTA。

## CTA 与转化承接

内容页不是只提供阅读，必须让用户知道下一步做什么。

必须做到：

- frontmatter 中优先使用 `parentToolSlug`、`seoCluster`、`primaryKeyword`、`pageIntent`、`conversionTarget`。
- CTA 文案写真实动作，例如“Generate a privacy policy draft”，不要写泛泛的 “Learn more”。
- CTA 位置不只在结尾。用户刚得到答案、读完步骤、看到边界或理解问题后，都可以出现轻量 CTA。
- CTA 不能打断快速答案；先回答问题，再引导下一步。

## 内容页视觉原则

内容页可以比工具页更克制，但不能没有视觉系统。

默认要求：

- 普通 cluster article 至少有产品化 Hero 外壳；需要图片时可用 1 张 Hero 图 + 1 到 2 张正文图。
- 高价值 hub / pillar / comparison / template 页面需要更强视觉，通常 1 张 Hero 图 + 2 到 4 张正文图。
- 短 FAQ / troubleshooting 可以少图，但仍需清晰 CTA、相关工具和边界。
- 图片必须服务某个段落、判断标准、示例或产品下一步，不能只是装饰。

内容页的图片要求见 `p0-asset-gates.md` 的“内容页视觉物料”章节。

## 禁止

- 只用一个窄 `main` 包住 H1、正文和 CTA。
- 把内容页做成没有品牌、没有 CTA、没有相关工具的裸文章。
- 每篇内容复制一套独立 HTML 或 React 布局，导致后续无法统一升级。
- 用工具页表单规则硬套内容页。
- 用图片承载 SEO 关键正文。
- 让 AI 每次生成全新的页面 UI 风格。
- 把阅读时间、分类、更新时间、关键词标签当作首屏核心卖点。

## 验收

交付前检查：

- [ ] 页面使用内容页共享 shell 或等价产品化外壳。
- [ ] 首屏能看出搜索问题、页面承诺和 Tomako 品牌关系。
- [ ] 前 20% 内容给出答案或路线，不绕背景。
- [ ] 正文有清晰层级、可读宽度和专业样式。
- [ ] CTA 指向真实 Tool / Template /下一步。
- [ ] 关键词、cluster、父工具和索引状态可从 frontmatter 或页面看出。
- [ ] 图片如存在，服务内容价值，不是装饰或信息复读。
- [ ] 移动端无横向溢出，侧栏和 CTA 顺序合理。
