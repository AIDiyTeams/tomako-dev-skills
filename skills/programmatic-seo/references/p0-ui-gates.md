# P0 UI 质量门槛

## 使用时机

涉及工具工作台、表单、按钮、tab、选择项、布局、组件库、移动端、结果区或页面视觉结构时，必须读取本文件。

本文件吸收并去重了原 `Tomako/docs/modules/tools-ui-style-guide.md` 中的 UI 层规则，并以用户最新要求为准处理冲突。

## 总原则

Tool 页面首先要让用户敢用、会用、能完成任务。UI 的目标不是展示复杂能力，而是降低决策压力。

必须做到：

- 表单简洁、自解释、视觉上不吓人。
- 工具本体靠近页面上方，不能藏在长营销内容后面。
- 重复控件以文字为主，少用 icon。
- 复杂表单不居中撑满页面。
- 页面主体不要被全局 `max-width` 收紧在中间。
- 结果区有足够阅读空间，不和表单挤在一个小卡里。
- 桌面和移动端都必须检查真实状态。

## 组件库优先

所有工具交互控件优先使用 `@/components/ui`：

- `Button`
- `Input`
- `Textarea`
- `Select` / `NativeSelect`
- `Checkbox`
- `RadioGroup`
- `Switch`
- `Tabs`
- `DropdownMenu`
- `Popover`
- `Dialog`
- `Card`
- `Badge`

不要在 Tool widget 内手写原生 `<select>`、checkbox、radio、tab、button、menu 的样式。若缺少组件，优先补共享组件，遵循现有 shadcn/Radix 模式。

## 表单复杂度

P0 要求：

- 表单必须通过 3 秒行动测试：用户不读长说明，也能看出要输入什么、点哪个按钮、等待什么结果。
- 表单区不是说明文档，不要在每个字段下面写大量小字。
- 不要给每个字段、标题、选项、tab、清单项都加 icon。
- 一个字段通常只需要 label、placeholder、必要校验；只有能避免真实错误时才加 helper。
- 表单顶部只保留任务名和最多一句必要说明，不要放段落式功能介绍。
- 产品教育、例子、流程解释、支持范围、失败边界、隐私/版权说明放到页面说明区、结果指导、FAQ 或折叠帮助里。
- 次要说明可以收进 `Tooltip`、`Popover` 或折叠帮助，触发点放在 label 或 section title 旁；不要给每个字段都加说明入口。
- 表单内不要放大块说明卡解释支持范围、浏览器转换、失败原因、隐私或版权；这些属于工作台下方的公开模块。
- 不要把所有选项铺成标签墙、chip 墙、pill 墙。
- 单选或大量选项优先用 `Select`、下拉、combobox、popover、折叠高级设置或分步流程。
- chip/pill 只适合少量、高信号、比下拉更容易扫读的选择。
- 第一屏只收集最少可用信息；高级信息可以折叠或交给 Agent 通过 `missingInfo` 返回。
- 工作台标题区只放真实任务动作或真实状态，不放和当前操作无关的装饰性 badge、chip 或 SEO 标签。

## 复杂表单布局

复杂表单默认采用：

```text
左侧：约束宽度的表单
右侧：低信息密度 before/after 视觉图
下方：独立生成结果区
```

规则：

- 表单不要放在页面正中并拉到很宽。
- 输入框、下拉框宽度要和真实输入长度匹配，不要因为容器宽就无限拉长。
- 右侧视觉图只表达“输入什么 -> 得到什么”，不承载真实结果。
- 真实生成结果必须在表单和视觉图下方，作为独立页面 section。
- 结果区不要和表单共享同一个大背景卡片。
- 用户点击生成后，必须把结果锚点滚到视觉中心，再进入等待或加载状态。
- 表单内主生成按钮必须横向拉通，纯文字，不加装饰 icon。

## 结果区

结果区必须根据输出形态给足空间：

- 长文档、报告、协议、候选列表、分阶段详情，默认使用下方整宽结果区，并吃满页面可用宽度。
- 不要用左表单右结果压缩长输出。
- 初始态不要保留大面积空结果面板，除非它直接帮助用户理解下一步。
- 结果区至少支持空、加载、等待、成功、错误、重试、复制、下载或导出。
- 多文档结果可以使用 tab、锚点、文档列表 + 阅读区，但要保持可读宽度。
- 重试或多轮修改时保留上一次有效结果。

结果区必须按当前工具任务设计状态，不要只放一个泛用“等待结果”空卡。

下载、转换、解析、提取类工具至少考虑：

- 初始态：轻量提示即可，不占用大面积视觉。
- 输入无效：说明链接、文件或格式哪里不对。
- 处理中：说明正在解析、转换或准备文件。
- 成功态：展示预览、格式、大小、下载/复制动作。
- 部分成功：展示可用结果和失败项。
- 失败态：说明可能原因，例如受限内容、格式不支持、网络或平台限制。

生成、分析、报告类工具至少考虑：

- loading / awaiting result。
- 缺失信息。
- schema 或结果无效。
- 可复制、下载、重试或继续修改。

## 页面宽度与流体布局

P0 要求：不要给整个 Tool 页面套一个固定的中间窄容器。工具页不是文章页，页面主体、图文模块、视觉 band、结果区和 CTA 默认应根据浏览器宽度流体适配，尽量使用左右可用空间。

必须做到：

- 不要用一个全局 `max-width: 1120px/1280px` 或类似容器包住所有内容。
- 桌面端左右留白要服务呼吸感，不要把内容挤成中间细条。
- 工具工作台、结果区、图文说明区、视觉物料区、CTA band 默认可以接近整屏可用宽度。
- 宽度限制只加在真正需要限制的对象上，例如左侧表单、单列长段落、长文档正文、FAQ 文本列。
- 表单可以有明确最大宽度，因为表单太宽会难填；但右侧视觉、下方结果和图文区域不要跟随表单一起被收窄。
- 长文本通过内部文本列控制可读行长，而不是收窄整个 section。
- 超宽屏可使用最小安全边距和响应式 grid，但不要让页面看起来像只占屏幕中间一小块。

禁止：

- 页面根节点、主内容壳、所有 section 共用同一个窄 `max-w-*`。
- 为了“整齐”把结果区、视觉区、FAQ、CTA 全部限制到表单宽度。
- 左右明明有大量空间，却让所有内容只占中间蓝框大小。
- 用文章页的排版逻辑约束工具页。

## 选中态与重复控件

P0 要求：

- tab、segmented control、chip、option pill 的选中态不能默认使用大面积纯黑。
- 选中态优先用浅强调底、强调边框、增强字重、轻阴影或局部色条。
- 近黑色只用于主 CTA、少量 badge、文字、边框或极少数强调点。
- 不要给每个选项都加 check icon；文本足够清楚时，用排版和状态色表达选择。
- 多个重复选项同时出现时，要让视觉重量轻，避免满屏按钮感。

## Icon 使用

可以使用 icon 的位置：

- 明确命令，例如复制、下载、删除、展开。
- 状态，例如成功、错误、警告、加载。
- 空状态或错误状态中帮助用户理解问题。
- 紧凑工具栏中能降低阅读成本的按钮。

不要使用 icon 的位置：

- 每个表单字段标题。
- 每个 tab。
- 每个选项。
- 每个卡片标题。
- 每个 checklist 行。
- 已经足够清楚的纯文字选择项。

## 色彩与层级

继承旧设计规范中有效部分，并按最新 P0 调整：

- 大面积背景优先白色、暖白、浅灰、极浅蓝灰或米白。
- 标题用近黑建立层级。
- 每个模块最多一个主强调色。
- 不要整页使用紫蓝 AI 渐变。
- 不要在同一个工作台里混用浅灰青背景、电蓝 icon、荧光绿 CTA。
- 浅色工作台不要紧贴大面积纯黑结果面板。
- 卡片、工具面板、图片与页面背景必须有可见边界，不能同色融在一起。
- 不要做页面背景 -> 大卡 -> section 卡 -> item 卡的多层嵌套。

## 字体、间距和卡片

建议：

- 桌面左右安全边距按屏幕宽度响应式变化，优先使用可用空间；移动端 20 到 24px。
- Hero H1 可以大，但不要和工具入口抢层级。
- Section 间距要有呼吸感，通常 72 到 160px。
- 工具工作台之后的说明区直接铺在页面背景上，避免连续套卡。
- 小卡片 padding 20 到 28px，大功能面板 28 到 40px。
- 说明文案用正常标题、段落、紧凑列表，不要伪装成 dropdown、accordion 或巨型 pill 卡。

## Hero 与页面结构

serious public Tool 页面需要 Hero 背景图。Hero 不再是左标题右卡片的普通两栏模块，而是一个用生图背景承载氛围和功能识别的首屏区域。

Hero 要求：

- 使用 `public/tools/{slug}-hero.png` 或等价 Hero 背景图。
- Hero 背景图铺满 Hero 区或首屏大区域，图片本身低信息密度但能点题。
- H1、intro、CTA、关键词和核心说明作为可抓取 HTML 文本覆盖在背景图上。
- Hero 图需要留出干净空间给文字覆盖，不能让画面主体压住标题。
- Hero 图内部如有文字，必须是英文短标签；页面覆盖文字按 locale 本地化。
- 首屏或首屏下沿能看到工具工作台入口，不要让 Hero 把工具入口推得太远。

简单 calculator/checklist/converter 也可以用紧凑 Hero，但仍需要背景图，只是高度、文字量和视觉复杂度可以更克制。

首屏右侧信息卡、metadata 卡或视觉卡必须帮助用户理解工具能力，例如支持格式、输入类型、输出类型、处理边界、隐私或更新时间。不要用“处理工具 / 生成器 / 5 min”这类弱信息占据首屏关键空间，除非它们确实影响用户决策。

不要把 Hero 做成大标题左侧、metadata/标签/阅读时间卡片右侧的左右对排。右侧小卡片不能替代 Hero 背景图，也不能替代“你可以得到什么 / 用户价值”模块。

不要把“你可以得到什么 / 用户价值”模块放在 Hero 大标题右侧做左右对排。Hero 的主任务是让用户确认页面任务；价值结果模块更适合放在首段说明之后、工具工作台附近、工作台下方、结果区之前或作为独立横向 band。

如果需要在首屏补充能力信息，优先使用对用户有决策价值的内容，例如：

- 可获得的结果类型。
- 支持和不支持的输入范围。
- 输出格式或导出方式。
- 失败时能得到什么提示。
- 隐私、版权、复核或平台边界。

不要用以下内容占据 Hero 右侧大面积空间：

- 阅读时间。
- 工具分类。
- 更新时间本身。
- 关键词标签。
- 泛泛 metadata。

更新时间可以保留为小号辅助信息，但不能作为核心信息卡。关键词变体应自然进入正文、H2、支持范围和 FAQ，不要用标签墙替代。

公开页面不要把内部方案评估做成主 section。若确实要解释“为什么不用 Agent / 为什么不用 LLM / 为什么是本地处理”，应改写成用户价值，例如更快、更私密、更低成本、无需等待，并放在能力边界或 FAQ 中轻量说明。

## Section 多样性

连续说明区不能机械复制同一个模板。

页面模块可以自由组织，不必固定成某几个标准模块。但除工具交互和基础使用说明外，每个模块都必须服务两个目标之一：

- 体现产品价值：展示核心能力、优势、差异化、输出质量、适用场景或工作流收益。
- 解决用户疑问：回答输入、输出、准确性、失败、隐私、边界、导出、复核、下一步等疑虑。

页面不能把“它是怎么工作的”当成默认主体结构。用户不关心内部流程，除非这个流程能直接降低疑虑或帮助完成任务。普通用户页面应优先展示：

- 为什么这个工具值得用。
- 一个好的同类工具应该具备什么。
- Tomako 的输出为什么更清楚、更可用或更放心。
- 结果拿到后可以怎样用。
- 常见失败、边界和责任说明。

如果左侧已经用文字说明价值或边界，右侧视觉不能再用 1、2、3、4 列表复读。右侧视觉必须补充情绪价值、专业感、结果预览、before/after 差异或可用性证明。

“好的标准 / 为什么选我们 / 用户价值”这类模块不能做成左侧编号清单、右侧大段正文的重复解释结构。用户会分不清左右关系，也会感觉信息很重。

更好的结构：

- 紧凑标准卡：每张卡只写一个标准，例如稳定、速度、质量、免费、易用、隐私。
- 对比表：左列是普通工具常见问题，右列是 Tomako 的处理方式。
- 单列说明：H2 + 3 到 5 个短块，每块标题 + 一句话。
- FAQ/边界说明：如果内容主要是失败原因、支持范围、版权隐私边界，可以放进 FAQ 或疑问消除模块。
- 视觉 + 文案：左侧放生图或场景图，右侧写标准；不要两边都写文字清单。

避免：

- 左边 1、2、3、4，右边再用长段落解释同一批点。
- 左右两边都在讲同一个标准，但没有明确对比、映射或层级。
- 把二级说明做成大段正文，导致用户扫不出结论。
- 用“过程透明”类内容占据“好标准”的核心位置。

可混排：

- 左文案 + 右视觉。
- 左视觉 + 右结果解读。
- 全宽视觉 band。
- editorial callout。
- 三列轻卡。
- before/after 对比。
- timeline/checklist。
- CTA band。

结构变化必须服务理解，不是为了装饰。连续 3 个以上相同两栏比例、相同列表形态、相同图片尺寸，是不合格。

如果一个模块既没有提升产品价值感，也没有减少用户疑虑，即使视觉好看也不应该保留。

## 移动端

必须检查：

- 标题不溢出，不超过合理行数。
- 表单单列可用。
- 下拉、按钮、checkbox、tab 可点击。
- 生成按钮满宽。
- 结果区不横向滚动。
- 图片主体不被裁到太小。
- 长文档阅读宽度和行高可读。

## UI 验收

交付前逐项检查：

- [ ] 使用了共享组件库，未手写默认浏览器控件。
- [ ] 表单通过 3 秒行动测试：不用读长说明也知道输入、动作和结果等待。
- [ ] 表单没有段落式功能介绍、大块说明卡、密集 helper、小字说明和 icon 装饰。
- [ ] 支持范围、失败边界、隐私/版权等说明已移到页面模块、FAQ、结果指导或轻量 tooltip/popover。
- [ ] 工作台标题区没有无关 badge、chip、SEO 标签或装饰按钮。
- [ ] 多选项没有铺成令人害怕的标签墙。
- [ ] 复杂表单为左表单、右视觉、下方结果。
- [ ] 生成按钮满宽、纯文字、无 icon。
- [ ] 点击生成会滚动到结果锚点中心。
- [ ] 选中态没有大面积纯黑。
- [ ] 重复控件没有给每项都加 icon/check。
- [ ] 初始态没有无意义大空结果区。
- [ ] 结果区能承载真实输出长度。
- [ ] 结果区状态贴合当前工具任务，覆盖无效输入、处理中、成功、失败和可操作结果。
- [ ] 按钮文案是用户动作，不是内部处理步骤。
- [ ] 首屏信息卡有真实决策价值，不是弱 metadata 占位。
- [ ] Hero 使用生图背景图，H1 / intro 作为 HTML 文本覆盖在图上。
- [ ] Hero 不是左标题右 metadata/标签/阅读时间卡片的两栏排版。
- [ ] “你可以得到什么 / 用户价值”模块没有和 Hero 大标题左右对排，而是在工作台附近或独立 section 中清楚呈现。
- [ ] 阅读时间、工具分类、更新时间、关键词标签没有占据首屏核心卡片位置。
- [ ] 页面没有把内部技术方案评估作为普通用户可见主模块。
- [ ] 说明模块不是固定模板堆叠，每个模块都服务产品价值或用户疑问。
- [ ] 价值模块让功能显得更专业、更有优势；疑问模块让用户更清楚、更放心地使用。
- [ ] 页面没有用“它是怎么工作的”替代“为什么值得用”和“为什么选我们”。
- [ ] 左右图文没有重复表达；视觉图不是正文旁边的 1234 流程复读。
- [ ] “好标准 / 为什么选我们 / 用户价值”模块不是左编号清单 + 右长段落的重复结构。
- [ ] 好标准模块优先呈现用户关心的结果标准，例如稳定、速度、质量、成本、易用、兼容、可信。
- [ ] 桌面和移动端真实状态都通过检查。
