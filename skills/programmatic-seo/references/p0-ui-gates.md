# P0 UI 质量门槛

## 使用时机

涉及工具工作台、表单、按钮、tab、选择项、布局、组件库、移动端、结果区或页面视觉结构时，必须读取本文件。

本文件吸收并去重了原 `Tomako/docs/modules/tools-ui-style-guide.md` 中的 UI 层规则，并以用户最新要求为准处理冲突。

## 总原则

Tool 页面首先要让用户敢用、会用、能完成任务。UI 的目标不是展示复杂能力，而是降低决策压力。

UI 不是页面信息架构的起点。完整页面优化或诊断必须先完成 `p0-content-gates.md` 中的 Copy Brief 和信息模块清单，再用 UI 把这些信息更清晰、更有吸引力地呈现出来。

必须做到：

- 表单简洁、自解释、视觉上不吓人。
- 工具本体靠近页面上方，不能藏在长营销内容后面。
- 重复控件以文字为主，少用 icon。
- 复杂表单不居中撑满页面。
- 页面主体不要被全局 `max-width` 收紧在中间。
- 结果区有足够阅读空间，不和表单挤在一个小卡里。
- Button 默认使用中性黑 / 近黑，不把工具主题色做成绿色或青色 CTA。
- 桌面和移动端都必须检查真实状态。

禁止：

- 先套 Hero、两栏、卡片、步骤、FAQ 或视觉模板，再倒推文案。
- 为了版式对称保留无价值模块，或把同一段信息拆成左右两边重复表达。
- 在没有确认用户价值、搜索需求和疑问覆盖前，只做 UI 美化并宣称页面优化完成。
- 用视觉图、metadata 卡、阅读时间、标签或流程图替代核心价值表达。

## 组件库优先

所有工具交互控件优先使用 `@/components/ui`：

- `Button`
- `Input`
- `Textarea`
- `Select` / `NativeSelect`
- `Checkbox`
- `RadioGroup`
- `Switch`
- `Tabs`
- `DropdownMenu`
- `Popover`
- `Dialog`
- `Card`
- `Badge`

不要在 Tool widget 内手写原生 `<select>`、checkbox、radio、tab、button、menu 的样式。若缺少组件，优先补共享组件，遵循现有 shadcn/Radix 模式。

## 共享模块使用原则

共享 section 和页面模板库是默认优先项，不是强制版式。页面必须先根据 Copy Brief 和信息模块清单判断要表达什么，再决定复用或自定义。

P0 要求：

- 能准确表达当前模块目标时，优先使用共享 section、共享外壳和 `@/components/ui`。
- 复用不能牺牲用户价值表达、疑问覆盖、结果阅读空间、表单简洁度或视觉质量。
- 如果共享模块导致信息重复、左右关系不清、模块价值弱、图片变成装饰、表单变复杂或结果区受限，必须自定义。
- 自定义模块也要遵守本文件的布局、宽度、色彩、表单、结果区、移动端和视觉规则。
- 新开发的自定义结构如果在两个以上页面稳定复现，再沉淀为共享组件。

交付前要能说明：这个页面哪些模块复用了模板库，哪些模块自定义，以及这样选择为什么更符合当前需求。

## 表单复杂度

P0 要求：

- 表单必须通过 3 秒行动测试：用户不读长说明，也能看出要输入什么、点哪个按钮、等待什么结果。
- 表单区不是说明文档，不要在每个字段下面写大量小字。
- 不要给每个字段、标题、选项、tab、清单项都加 icon。
- 一个字段通常只需要 label、placeholder、必要校验；只有能避免真实错误时才加 helper。
- 表单顶部只保留任务名和最多一句必要说明，不要放段落式功能介绍。
- 产品教育、例子、流程解释、支持范围、失败边界、隐私/版权说明放到页面说明区、结果指导、FAQ 或折叠帮助里。
- 次要说明可以收进 `Tooltip`、`Popover` 或折叠帮助，触发点放在 label 或 section title 旁；不要给每个字段都加说明入口。
- 表单内不要放大块说明卡解释支持范围、浏览器转换、失败原因、隐私或版权；这些属于工作台下方的公开模块。
- 不要把所有选项铺成标签墙、chip 墙、pill 墙。
- 单选或大量选项优先用 `Select`、下拉、combobox、popover、折叠高级设置或分步流程。
- chip/pill 只适合少量、高信号、比下拉更容易扫读的选择。
- 第一屏只收集最少可用信息；高级信息可以折叠或交给 Agent 通过 `missingInfo` 返回。
- 外露配置项只放最核心、最关键、最影响结果质量的字段，尤其是必填字段。
- 外露配置项最多 5 个；超过 5 个时，保留必要字段，其余默认收进“高级设置”“更多选项”或分步骤流程。
- “配置项”包括输入框、文本域、下拉、单选、多选、开关和文件/URL 上传，不包括主按钮、结果区或说明文案。
- 折叠项可以影响结果细节，但不能成为用户第一次使用前必须理解的一长串设置。
- 工作台标题区只放真实任务动作或真实状态，不放和当前操作无关的装饰性 badge、chip 或 SEO 标签。
- 工具工作台里的表单面板、控制区和工具栏默认使用白色背景；不要用大面积浅绿、浅蓝、浅灰等 tinted 底色承载整块输入区。
- 浅色强调底只用于选中态、提示、上传预览、空状态或局部信息块，不作为整个工具栏/表单面板的默认背景。
- 表单内主按钮、下载按钮、生成按钮和确认按钮默认使用黑色或近黑背景 + 白色文字；不要用绿色、青色、荧光色或页面主题色承载核心 CTA。
- 次要按钮用中性 outline 或黑色按钮，不要继续沿用绿色/青色 hover、border 或文字色。

## 复杂表单布局

复杂表单默认采用：

```text
左侧：约束宽度的表单
右侧：低信息密度 before/after 视觉图
下方：独立生成结果区
```

规则：

- 表单不要放在页面正中并拉到很宽。
- 输入框、下拉框宽度要和真实输入长度匹配，不要因为容器宽就无限拉长。
- 右侧视觉必须使用 `p0-asset-gates.md` 中定义的工作台右侧生图物料，不能用前端代码画一个示意图顶替。
- 右侧视觉图只表达“简单输入 -> 精美结果”，不承载真实结果。
- 真实生成结果必须在表单和视觉图下方，作为独立页面 section。
- 结果区不要和表单共享同一个大背景卡片。
- 用户点击生成后，必须把结果锚点滚到视觉中心，再进入等待或加载状态。
- 表单内主生成按钮必须横向拉通，纯文字，不加装饰 icon。

## 容器边界与横向溢出

表单、输入框、下拉、文本域、风格选择器、Type 选择器、横向卡片轨道、右侧视觉和结果区都必须被限制在自己的容器边界内。任何控件越过父容器、压到相邻列、盖住 textarea，或让整页出现意外横向滚动，都是 P0 错误。

P0 要求：

- 页面级 `body` / `main` 不允许因为 Tool UI 出现横向滚动。唯一允许的横向滚动必须发生在明确设计的内部滚动容器中。
- grid / flex 的可收缩子项必须设置能收缩的约束，例如 `min-w-0`、`max-w-full`、`overflow-hidden` 或内部 `overflow-x-auto`，不能让固定宽度子项把父容器撑破。
- `Input`、`Textarea`、`Select`、button、option card、result card、图片和横向轨道默认 `max-w-full`，不能因为 placeholder、长词、图片、卡片宽度或固定 `min-width` 撑出表单列。
- 大型风格选择器、Type 选择器、模板选择器、视觉预设选择器不应直接铺成超宽横向卡片墙。超过 3 到 4 个大卡选项时，优先改成 `Select`、popover、dialog、折叠高级设置或分步选择。
- 如果确实需要横向滚动选择器，滚动轨道必须在表单列内部自洽：父级 `max-w-full`，轨道 `overflow-x-auto`，卡片宽度使用响应式 `clamp` / `basis`，不能越过相邻右侧视觉或覆盖下面字段。
- “可左右滑动”提示不能替代布局约束。提示存在时，也必须保证滑动区域不撑破页面、不遮挡其他控件。
- 右侧工作台视觉、图片或 absolute 装饰不能覆盖左侧表单。两栏布局的左右列都必须有边界约束；必要时降低右侧视觉层级、改成普通文档流或在窄宽度下上下堆叠。
- Popover / dropdown / dialog 可以临时浮层展示，但不能挡住当前输入路径，也不能让页面出现永久横向溢出。

实现时优先检查这些低级原因：

- grid 子项缺 `min-w-0`。
- 横向列表用了固定 `min-width`，但父级没有 `overflow-x-auto`。
- 表单列或外层 panel 没有 `max-w-full`。
- 图片/卡片使用固定宽度或绝对定位，脱离正常布局。
- textarea、input、placeholder 或长中文/英文没有在容器内处理。

## 结果区

结果区必须根据输出形态给足空间：

- 长文档、报告、协议、候选列表、分阶段详情，默认使用下方整宽结果区，并吃满页面可用宽度。
- 不要用左表单右结果压缩长输出。
- 初始态不要保留大面积空结果面板，除非它直接帮助用户理解下一步。
- 结果区至少支持空、加载、等待、成功、错误、重试、复制、下载或导出。
- 多文档结果可以使用 tab、锚点、文档列表 + 阅读区，但要保持可读宽度。
- 重试或多轮修改时保留上一次有效结果。

结果区必须按当前工具任务设计状态，不要只放一个泛用“等待结果”空卡。

下载、转换、解析、提取类工具至少考虑：

- 初始态：轻量提示即可，不占用大面积视觉。
- 输入无效：说明链接、文件或格式哪里不对。
- 处理中：说明正在解析、转换或准备文件。
- 成功态：展示预览、格式、大小、下载/复制动作。
- 部分成功：展示可用结果和失败项。
- 失败态：说明可能原因，例如受限内容、格式不支持、网络或平台限制。

生成、分析、报告类工具至少考虑：

- loading / awaiting result。
- 缺失信息。
- schema 或结果无效。
- 可复制、下载、重试或继续修改。

## 长报告加载体验

长报告型 Agent Tool 不能只用一个全局大 loading 等 final result。用户应该能顺着页面看到真实结果逐步出现，并清楚知道下一块是否还在生成。

P0 要求：

- 已完成模块立即稳定展示，不被后续 loading、timeout 或重试状态覆盖。
- 下一个模块的占位和 loading 必须出现在内容流中的真实位置，不能让用户滑到一片空白。
- 顶部 overall progress card 可以保留，用于说明整体任务状态；局部 module loading 用于说明当前正在生成哪个模块，两者职责不同，可以同时存在。
- 模块 loading 文案说明当前模块名、步骤或阶段，例如 `step X/N`；不要只写泛泛“正在分析”。
- 时间估计必须单调：剩余时间不能突然跳回更长；倒计时到 0 后改成“仍在处理”，不要循环重置。
- 有 partial 时，超时状态应表达为“继续生成 / 已保留当前结果 / 可稍后重试”，而不是整页失败。
- 不要用假进度、前端 mock partial 或固定 demo 模块冒充真实 Agent 输出。

## 页面宽度与流体布局

P0 要求：不要给整个 Tool 页面套一个固定的中间窄容器。工具页不是文章页，页面主体、图文模块、视觉 band、结果区和 CTA 默认应根据浏览器宽度流体适配，尽量使用左右可用空间。

必须做到：

- 不要用一个全局 `max-width: 1120px/1280px` 或类似容器包住所有内容。
- 桌面端左右留白要服务呼吸感，不要把内容挤成中间细条。
- 工具工作台、结果区、图文说明区、视觉物料区、CTA band 默认可以接近整屏可用宽度。
- 宽度限制只加在真正需要限制的对象上，例如左侧表单、单列长段落、长文档正文、FAQ 文本列。
- 表单可以有明确最大宽度，因为表单太宽会难填；但右侧视觉、下方结果和图文区域不要跟随表单一起被收窄。
- 长文本通过内部文本列控制可读行长，而不是收窄整个 section。
- 超宽屏可使用最小安全边距和响应式 grid，但不要让页面看起来像只占屏幕中间一小块。

禁止：

- 页面根节点、主内容壳、所有 section 共用同一个窄 `max-w-*`。
- 为了“整齐”把结果区、视觉区、FAQ、CTA 全部限制到表单宽度。
- 左右明明有大量空间，却让所有内容只占中间蓝框大小。
- 用文章页的排版逻辑约束工具页。

## 选中态与重复控件

P0 要求：

- tab、segmented control、chip、option pill 的选中态不能默认使用大面积纯黑。
- 选中态必须一眼可见，不能只靠轻微底色变化、1px 浅边框或 hover 同级变化表达。
- 选中态至少使用两种可见信号组合，例如中性浅灰底、2px 近黑描边、增强字重、轻阴影或图标容器反色。
- option card / chip / 结果卡这类小选择项的 active 态只保留一个外轮廓；不要同时叠加 border + ring 形成双描边。
- option card / chip 这类小选择项不要额外加凸起色块、顶部标签条或悬浮装饰块；背景 + 2px 描边通常已经足够。
- 选中态、hover 态、focus 态和普通态必须能明显区分；键盘 focus 也要有可见 ring。
- 近黑色只用于主 CTA、少量 badge、文字、边框或极少数强调点。
- 不要给每个选项都加 check icon；文本足够清楚时，用排版和状态色表达选择。
- 多个重复选项同时出现时，要让视觉重量轻，避免满屏按钮感。

## Icon 使用

可以使用 icon 的位置：

- 明确命令，例如复制、下载、删除、展开。
- 状态，例如成功、错误、警告、加载。
- 空状态或错误状态中帮助用户理解问题。
- 紧凑工具栏中能降低阅读成本的按钮。

不要使用 icon 的位置：

- 每个表单字段标题。
- 每个 tab。
- 每个选项。
- 每个卡片标题。
- 每个 checklist 行。
- 已经足够清楚的纯文字选择项。

## 色彩与层级

继承旧设计规范中有效部分，并按最新 P0 调整：

- 大面积背景优先白色、暖白、浅灰或米白。
- 工具栏、表单控制区和工作台左侧输入面板优先白底，通过边框、间距和局部状态色建立层级。
- 标题用近黑建立层级。
- Tool UI 默认使用中性灰黑体系；不要把绿色、青色或蓝绿色作为默认工具主题色。
- 每个模块最多一个主强调色。强调色优先服务状态、风险、平台品牌或图片内容，不作为整页控件主题。
- Button / CTA 默认黑色或近黑；除非已有明确品牌系统或业务语义，否则不要使用绿色、青色、橙色或渐变按钮。
- 不要整页使用紫蓝 AI 渐变。
- 不要在同一个工作台里混用浅灰青背景、电蓝 icon、荧光绿 CTA 或其他强主题色组合。
- 浅色工作台不要紧贴大面积纯黑结果面板。
- 卡片、工具面板、图片与页面背景必须有可见边界，不能同色融在一起。
- 不要做页面背景 -> 大卡 -> section 卡 -> item 卡的多层嵌套。

## 字体、间距和卡片

建议：

- 桌面左右安全边距按屏幕宽度响应式变化，优先使用可用空间；移动端 20 到 24px。
- Hero H1 可以大，但不要和工具入口抢层级。
- Section 间距要有呼吸感，通常 72 到 160px。
- 工具工作台之后的说明区直接铺在页面背景上，避免连续套卡。
- 小卡片 padding 20 到 28px，大功能面板 28 到 40px。
- 说明文案用正常标题、段落、紧凑列表，不要伪装成 dropdown、accordion 或巨型 pill 卡。

## Hero 与页面结构

serious public Tool 页面需要 Hero 背景图。Hero 不再是左标题右卡片的普通两栏模块，而是一个用生图背景承载氛围和功能识别的首屏区域。

## SEO 导航栏

Tool/SEO 页面顶部导航必须和当前本地前端源码里的官网 Hero 导航保持一致，默认复用共享导航组件，而不是在 Tool 页里手写“返回工具”专用导航。

要求：

- 使用当前前端库和本地源码中的官网导航实现；不要以线上已部署页面截图或旧 `MarketingHeader` 为准。
- 导航栏使用 `@/components/ui/navigation-menu` 承载主导航链接。
- Tool/SEO 页导航必须有独立背景层，默认使用共享导航的 solid surface，确保压在 Hero 背景图或页面图像上时仍清晰可读。
- 官网首页 Hero 可以保持透明导航；不要为了修 Tool/SEO 页导航背景而改首页 Hero 的导航视觉。
- 最左侧 Tomako logo 在 Tool 页点击后跳转到当前 locale 的官网首页，例如 `/zh` 或 `/en`。
- 产品、定价等导航项在 Tool 页跳转到官网首页对应锚点，例如 `/zh#workflow`、`/zh#pricing`；首页自身可以继续使用 `#workflow`、`#pricing`。
- Tool 页导航不需要当前页选中态，不要因为 SEO 页面 slug、工具分类或当前位置额外加 active chip、pill、返回按钮或分类标签。
- 不要把 MCP badge、阅读时间、更新时间、工具分类、关键词、输入/输出/边界三标签塞进导航栏。必要的 MCP 入口放在 Hero 文案区或工具说明区，不能破坏官网导航结构。

Hero 要求：

- 使用 `public/tools/{slug}-hero.png` 或等价 Hero 配图。
- Hero 配图必须是透明背景 PNG，不要生成或导入一整张带背景色的大图。
- Hero 区域自身背景色保持 `#F7F6F2`；透明 PNG 直接透出 Hero 背景色，避免右侧出现独立色块、背景板或割裂感。
- Hero 配图仍保留共享 shell 展示逻辑：在 1200px 容器范围内居右，使用 `object-contain object-right`。
- 不要把 Hero 配图改成 `object-cover` 或全屏铺满，避免大屏下无限放大、主体被裁切或透明图变成带底色背景。
- Hero 配图本身低信息密度但能点题。
- H1、intro、CTA、关键词和核心说明作为可抓取 HTML 文本覆盖在背景图上。
- Hero 图需要留出干净空间给文字覆盖，不能让画面主体压住标题。
- 桌面端 Hero 背景图的主视觉默认靠右呈现，左侧作为文字安全区；可通过 `object-position: right center`、响应式裁切或局部遮罩保证主视觉不被文字压住。
- 左侧文字区可以叠加极轻的装饰纹理或远景元素，但不能出现重要物件、可读标签、结果卡、强对比线条或新的视觉焦点。
- 不要为了避免遮挡把整张图整体缩小到远处。优先通过构图、定位和安全区解决遮挡，保持右侧主体足够大、足够有价值感。
- Hero 图内部如有文字，必须是英文短标签；页面覆盖文字按 locale 本地化。
- 首屏或首屏下沿能看到工具工作台入口，不要让 Hero 把工具入口推得太远。

简单 calculator/checklist/converter 也可以用紧凑 Hero，但仍需要背景图，只是高度、文字量和视觉复杂度可以更克制。

不要把 Hero 做成大标题左侧、metadata/标签/阅读时间卡片右侧的左右对排。右侧小卡片不能替代 Hero 背景图，也不能替代“你可以得到什么 / 用户价值”模块。

不要把“你可以得到什么 / 用户价值”模块放在 Hero 大标题右侧做左右对排。Hero 的主任务是让用户确认页面任务；价值结果模块更适合放在首段说明之后、工具工作台附近、工作台下方、结果区之前或作为独立横向 band。

不要用以下内容占据 Hero 右侧大面积空间：

- 阅读时间。
- 工具分类。
- 更新时间本身。
- 关键词标签。
- 泛泛 metadata。
- 自动生成的输入 / 输出 / 边界三标签。

更新时间可以保留为小号辅助信息，但不能作为核心信息卡。关键词变体应自然进入正文、H2、支持范围和 FAQ，不要用标签墙替代。

输入、输出、边界是开工前需求建模信息，不是默认 Hero UI。除非它们本身就是当前工具的核心交互控件或高价值判断模块，否则不要在 Hero 下方或右侧自动渲染为三张 pill / signal card。需要说明输入范围、输出内容、隐私、失败边界或复核责任时，应放到工作台附近的帮助、Result guidance、能力边界、FAQ 或结果区状态中，用用户价值语言表达。

公开页面不要把内部方案评估做成主 section。若确实要解释“为什么不用 Agent / 为什么不用 LLM / 为什么是本地处理”，应改写成用户价值，例如更快、更私密、更低成本、无需等待，并放在能力边界或 FAQ 中轻量说明。

## Section 多样性

连续说明区不能机械复制同一个模板。

页面模块可以自由组织，不必固定成某几个标准模块。但除工具交互和基础使用说明外，每个模块都必须服务两个目标之一：

- 体现产品价值：展示核心能力、优势、差异化、输出质量、适用场景或工作流收益。
- 解决用户疑问：回答输入、输出、准确性、失败、隐私、边界、导出、复核、下一步等疑虑。

页面不能把“它是怎么工作的”当成默认主体结构。用户不关心内部流程，除非这个流程能直接降低疑虑或帮助完成任务。普通用户页面应优先展示：

- 为什么这个工具值得用。
- 一个好的同类工具应该具备什么。
- Tomako 的输出为什么更清楚、更可用或更放心。
- 结果拿到后可以怎样用。
- 常见失败、边界和责任说明。

如果左侧已经用文字说明价值或边界，右侧视觉不能再用 1、2、3、4 列表复读。右侧视觉必须补充情绪价值、专业感、结果预览、before/after 差异或可用性证明。

“好的标准 / 为什么选我们 / 用户价值”这类模块不能做成左侧编号清单、右侧大段正文的重复解释结构。用户会分不清左右关系，也会感觉信息很重。

更好的结构：

- 紧凑标准卡：每张卡只写一个标准，例如稳定、速度、质量、免费、易用、隐私。
- 对比表：左列是普通工具常见问题，右列是 Tomako 的处理方式。
- 单列说明：H2 + 3 到 5 个短块，每块标题 + 一句话。
- FAQ/边界说明：如果内容主要是失败原因、支持范围、版权隐私边界，可以放进 FAQ 或疑问消除模块。
- 视觉 + 文案：左侧放生图或场景图，右侧写标准；不要两边都写文字清单。

避免：

- 左边 1、2、3、4，右边再用长段落解释同一批点。
- 左右两边都在讲同一个标准，但没有明确对比、映射或层级。
- 把二级说明做成大段正文，导致用户扫不出结论。
- 用“过程透明”类内容占据“好标准”的核心位置。

可混排：

- 左文案 + 右视觉。
- 左视觉 + 右结果解读。
- 全宽视觉 band。
- editorial callout。
- 三列轻卡。
- before/after 对比。
- timeline/checklist。
- CTA band。

结构变化必须服务理解，不是为了装饰。连续 3 个以上相同两栏比例、相同列表形态、相同图片尺寸，是不合格。

如果一个模块既没有提升产品价值感，也没有减少用户疑虑，即使视觉好看也不应该保留。

## 移动端

必须检查：

- 标题不溢出，不超过合理行数。
- 表单单列可用。
- 下拉、按钮、checkbox、tab 可点击。
- 生成按钮满宽。
- 结果区不横向滚动。
- 图片主体不被裁到太小。
- 长文档阅读宽度和行高可读。

## UI 验收

交付前逐项检查：

- [ ] 使用了共享组件库，未手写默认浏览器控件。
- [ ] 已评估共享 section / 页面模板库；适合的模块已复用，不适合的模块已自定义并说明原因。
- [ ] 表单通过 3 秒行动测试：不用读长说明也知道输入、动作和结果等待。
- [ ] 表单没有段落式功能介绍、大块说明卡、密集 helper、小字说明和 icon 装饰。
- [ ] 支持范围、失败边界、隐私/版权等说明已移到页面模块、FAQ、结果指导或轻量 tooltip/popover。
- [ ] 工作台标题区没有无关 badge、chip、SEO 标签或装饰按钮。
- [ ] Button / CTA 使用黑色或近黑背景，次要按钮使用中性样式；没有默认绿色、青色、荧光色或渐变按钮。
- [ ] 工具 UI 主题为中性灰黑体系，绿色/青色没有作为表单、按钮、选中态、边框或 hover 的默认主题色。
- [ ] 外露配置项不超过 5 个，只保留核心必填或关键字段；其余已折叠或分步。
- [ ] 多选项没有铺成令人害怕的标签墙。
- [ ] 复杂表单为左表单、右视觉、下方结果。
- [ ] 输入框、textarea、select、风格/Type 选择器、横向卡片轨道、右侧视觉和结果区没有越过父容器、压住相邻列或遮挡后续字段。
- [ ] 页面没有意外横向滚动；如有横向选择器，滚动只发生在该内部轨道内，且轨道本身 `max-w-full`。
- [ ] 大型风格/模板/Type 选择器超过 3 到 4 个大卡选项时，已使用下拉、popover、dialog、折叠高级设置或分步选择，而不是超宽卡片墙。
- [ ] 生成按钮满宽、纯文字、无 icon，且使用黑色或近黑背景。
- [ ] 点击生成会滚动到结果锚点中心。
- [ ] 选中态没有大面积纯黑。
- [ ] tab、chip、option、结果卡等选中态一眼可见，至少有两种状态信号，且和 hover / focus / 普通态明显区分。
- [ ] 重复控件没有给每项都加 icon/check。
- [ ] 初始态没有无意义大空结果区。
- [ ] 结果区能承载真实输出长度。
- [ ] 结果区状态贴合当前工具任务，覆盖无效输入、处理中、成功、失败和可操作结果。
- [ ] 长报告工具已有模块稳定展示，下一个模块在内容流中提前显示占位和局部 loading；有 partial 时不会被 final timeout 覆盖成整页失败。
- [ ] 长报告 loading 的顶部总进度和局部模块进度职责清楚，时间估计不跳回更长剩余时间。
- [ ] 按钮文案是用户动作，不是内部处理步骤。
- [ ] 首屏信息卡有真实决策价值，不是弱 metadata 占位。
- [ ] Hero 使用透明背景 PNG 配图，Hero 区域背景色为 `#F7F6F2`，H1 / intro 作为 HTML 文本覆盖在图上。
- [ ] Hero 配图在 1200px 容器范围内居右，使用 `object-contain object-right`，没有带背景色整图、独立色块或 `object-cover` 裁切。
- [ ] Hero 主视觉靠右且不被左侧文案遮挡；左侧文字安全区干净，装饰不抢焦点。
- [ ] Hero 不是左标题右 metadata/标签/阅读时间卡片的两栏排版。
- [ ] “你可以得到什么 / 用户价值”模块没有和 Hero 大标题左右对排，而是在工作台附近或独立 section 中清楚呈现。
- [ ] 阅读时间、工具分类、更新时间、关键词标签没有占据首屏核心卡片位置。
- [ ] Hero 没有默认展示输入 / 输出 / 边界三标签；这些信息如有必要，已转成工作台帮助、结果指导、能力边界或 FAQ。
- [ ] 页面没有把内部技术方案评估作为普通用户可见主模块。
- [ ] 说明模块不是固定模板堆叠，每个模块都服务产品价值或用户疑问。
- [ ] 价值模块让功能显得更专业、更有优势；疑问模块让用户更清楚、更放心地使用。
- [ ] 页面没有用“它是怎么工作的”替代“为什么值得用”和“为什么选我们”。
- [ ] 左右图文没有重复表达；视觉图不是正文旁边的 1234 流程复读。
- [ ] “好标准 / 为什么选我们 / 用户价值”模块不是左编号清单 + 右长段落的重复结构。
- [ ] 好标准模块优先呈现用户关心的结果标准，例如稳定、速度、质量、成本、易用、兼容、可信。
- [ ] 桌面和移动端真实状态都通过检查。
