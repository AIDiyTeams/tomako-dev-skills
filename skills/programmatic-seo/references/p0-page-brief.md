# P0 开工 Brief Gate

## 使用时机

新建、重做或大幅调整任何 Tomako Tool/SEO 页面前，先读本文件。目标是把需求、输入、输出、搜索意图和风险边界说清楚到“足够开工”，而不是把用户拖进冗长 PRD。

## 开工前最低确认

必须确认：

- 目标用户：谁会搜索或使用这个工具。
- 搜索意图：用户为什么搜它，想完成什么任务。
- 用户输入：用户需要提供 URL、文件、文本、产品名、需求描述、选项，还是已有账号数据。
- 有用输出：页面或 Agent 最终返回什么，用户拿到后能做什么。
- 你可以得到什么：用户完成操作后能拿到的具体结果项，例如文件、报告、草案、清单、建议、链接、下载项或失败原因提示。
- 用户价值：这些结果能帮用户节省什么、避免什么、判断什么、交付什么或继续完成什么工作。
- 输出标准：草案、报告、评分、候选列表、清单、文件、建议、可下载内容，还是可继续编辑的结果。
- 为什么选我们：用户为什么应该使用 Tomako，而不是手工、模板、脚本、插件、普通文章或竞品。
- 好工具标准：同类工具怎样才算好，优先从用户决策标准写，例如稳定、速度、质量、成本、易用、兼容、可信，而不是只写流程透明。
- 搜索需求覆盖：核心词、同义变体、输入/输出词、失败排查词、边界词、结果使用词和下一步词。
- 支持与不支持：这个工具能处理什么，明确不处理什么。
- 失败与恢复：用户拿不到结果时如何排查、重试或换下一步。
- 结果使用：用户拿到结果后如何复制、下载、导出、复核、发布或继续编辑。
- 视觉物料计划：至少 5 张生图，包含 1 张 Hero 背景图、1 张工作台右侧图和 3 张正文配图；图片内部文字必须是英文短标签。
- 运行时倾向：纯前端、MCP、云端 Agent/LLM Task/Skills-OL，或混合流程。
- 风险边界：是否涉及法律、隐私、合规、财务、安全、医疗、商业承诺、用户数据或公开抓取。
- 发布目标：本地验证、review、内部试用、公开 SEO 页面，还是可索引生产页面。

## 输入输出优先原则

输入和输出是最重要的信息。用户可以只给粗糙想法，但 AI 必须主动把它整理成：

```text
页面名称：
目标用户：
主要搜索意图：
用户输入：
生成/分析过程：
有用输出：
你可以得到什么：
用户价值：
结果验收标准：
为什么选我们：
好工具标准：
替代方案/竞品对比：
搜索需求覆盖：
支持范围：
不支持范围：
失败排查：
结果使用指导：
相关工具/下一步：
视觉物料计划：
Hero 背景图：
正文配图 A/B/C：
需要后端/Agent 的理由：
风险与限制：
仍需确认的问题：
```

不要限制维度。不同工具可以需要完全不同的输入输出结构。判断依据是用户任务，而不是模板字段。

## 工具类型与用户任务校准

开工前必须先判断页面真实任务类型，不能只按内部实现或现成模板套壳。

必须确认：

- 用户动词：下载、解析、转换、生成、分析、评分、检查、压缩、提取、导出，哪一个是用户真正要做的动作。
- 输入对象：链接、文件、文本、图片、产品信息、业务 brief、账号数据，必须具体。
- 输出对象：下载文件、格式选项、媒体预览、报告、草案、候选列表、风险清单，必须具体。
- 工具类型：下载器、转换器、解析器、生成器、分析器、检查器，不要因为页面模板叫 generator 就把所有工具都写成生成器。
- 用户验收：用户怎样判断“这件事完成了”，例如拿到可下载文件、看到媒体预览、获得可复制草案、得到可执行清单。

如果一个工具的用户任务是下载、转换、解析或提取，页面 Brief 中不要把“是否使用 Agent / 是否使用 LLM / 技术方案评估”当作主价值。内部方案只影响实现，不应替代用户任务。

## 什么时候必须追问

出现以下情况，必须先问 2 到 5 个短问题，或明确写出假设：

- 不知道用户到底输入什么。
- 不知道结果应该是什么形态。
- 工具名像生成器、分析器、报告、协议、计划、推荐，但没有说明真实输出标准。
- 可能需要抓 URL、读产品信息、分析上下文、调用 Agent 或生成长文档。
- 可能涉及法律、隐私、合规、财务、安全等高风险领域。
- 用户提到“直接实现”，但架构选择会因答案不同而明显变化。

好的追问示例：

- “用户第一步主要输入产品 URL，还是输入产品描述？”
- “结果希望是一份完整文档，还是分段草案加风险清单？”
- “这版要接真实 Agent，还是先做 review 环境验证但不公开索引？”
- “输出需要支持复制、下载，还是后续多轮修改？”

## 不要过度确认

不要在第一轮索要完整法务、营销、技术或合规清单。高上下文工具应先收集最少可用信息，让 Agent 生成第一版结果，再把未知项作为 `missingInfo`、风险提示、复核清单或后续问题返回。

可以接受粗糙输入，但不能接受不清楚输出。

## 页面价值判断

开工前判断这个页面是否值得独立发布：

- 它是否解决一个独立用户任务，而不是只换关键词。
- 如果没有其他同类页面，它是否仍然有发布价值。
- 工具工作台是否是页面核心，而不是文章里塞一个表单。
- 页面是否比普通模板、博客、关键词页更有用。
- 页面是否清楚告诉用户“你可以得到什么”和“这些结果有什么价值”，而不是用阅读时间、工具分类、更新时间或标签替代。
- 页面是否讲清用户为什么选 Tomako，而不是只介绍流程。
- 页面是否定义了用户真正关心的“好工具标准”，例如稳定、速度、质量、成本、易用、兼容、可信，并把产品亮点放在这个标准中证明。
- 页面是否覆盖核心词、长尾词、失败排查、能力边界、结果使用和下一步，而不是只覆盖主关键词。
- 这个页面和已有工具是否重叠；若重叠，是否应该合并、改名或加 alias。
- 页面中的每个公开模块是否服务用户任务。若某个模块主要解释内部方案、技术取舍或 Agent 是否参与，默认不放在公开 SEO 页面主体里，除非目标用户明确需要这些信息。
- 第三方平台、公开内容、下载、抓取或转换类工具是否说明可处理范围、失败边界、版权/隐私/平台限制和用户责任。

## 运行时初判

默认需要云端 Agent/LLM Task/Skills-OL 的情况：

- 需要用户产品 URL、官网、App 链接、品牌信息或业务上下文。
- 需要爬取、分析、总结、生成、推荐、评分、报告或长文档。
- 需要创意判断、定位判断、合规判断、商业策略判断。
- 输出会因不同用户背景明显变化。
- 结果需要结构化写回给前端。

可以考虑纯前端的情况：

- 明确是低风险、确定性、自包含逻辑。
- 浏览器中可以可靠完成结果。
- 页面明确标注为本地 demo 或模板工具。
- 不会让用户误以为有真实 AI、抓取、法律、财务、安全或专业审核能力。

## Brief 交付要求

开始实现前，AI 应在对话中给出简短 Brief。它可以很短，但必须覆盖输入、输出、运行时和风险。若用户没有回答关键问题，必须说明采用了哪些假设。
