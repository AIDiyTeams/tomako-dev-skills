# P0 文案与 SEO 内容门槛

## 使用时机

涉及 SEO 文案、H1、metadata、FAQ、页面模块、可搜索性、可抓取内容、能力边界、内部链接或多语言文案时，必须读取本文件。

本文件合并了旧设计规范中的 SEO 与文案要求，并吸收外部 SEO playbook 的可用检查点。外部资料只能作为审查视角，不能覆盖 Tomako 架构和真实运行能力。

## SEO 工具页的内容目标

程序化 SEO 不是批量换关键词，而是用可复用系统生产真正有用、可抓取、可索引、能解决用户任务的页面。

SEO 工具页的模块可以自由组织，不需要固定套用某几个 section。但除工具交互和基础使用说明外，页面主体必须服务两个核心目标：

1. **体现产品价值**：把功能讲得足够漂亮、具体、可信，突出核心能力、优势、差异化和用户拿到的好处。
2. **解决用户疑问**：预判用户使用前、使用中、使用后的疑虑，并用清楚、低成本、让人放心的方式回答。

每个页面必须回答：

- 用户为什么搜索这个工具。
- 用户要输入什么。
- 工具会做什么。
- 用户会得到什么。
- 结果应该怎么使用。
- 工具有哪些限制。
- 这个工具为什么值得用，而不是随便找一个模板、脚本或竞品。
- 这个工具相比替代方案更好在哪里，用户为什么应该选择 Tomako。
- 用户可能担心什么，页面如何让他放心继续操作。

## 内容先行与信息模块清单

新建、重做、优化或诊断 SEO Tool 页面时，必须先完成内容判断，再进入 UI。页面不是先选排版模板再填文案，而是先从用户视角和 SEO 增长视角判断“这个页面必须回答什么、证明什么、消除什么疑虑”。

UI 设计前必须先列出信息模块清单。格式可以简短，但必须覆盖：

```text
模块名称：
用户问题：
SEO / 搜索意图：
核心信息：
产品价值或用户疑虑：
期望用户动作：
适合位置：
UI 呈现建议：
```

执行顺序：

1. 先判断用户搜索这个工具时真正想完成的任务。
2. 再列出必须出现的信息：输入、输出、可获得结果、价值、为什么选我们、好工具标准、支持范围、不支持范围、失败排查、结果使用、相关工具。
3. 再判断每个信息模块属于价值型、疑问型还是操作型。
4. 再安排模块优先级和页面位置。
5. 最后才进入 UI、配图、布局和组件选择。

P0 禁止：

- 先画 Hero、卡片、两栏、FAQ、步骤图，再回头给这些容器补文案。
- 因为某个 UI 模板好看，就保留没有用户价值或搜索价值的模块。
- 把 UI 排版问题误当成文案问题，只调样式不补用户价值、搜索需求或疑问解答。
- 让视觉图、metadata 卡或流程卡替代“你可以得到什么”“为什么选我们”“支持与失败边界”等核心信息。

如果当前页面诊断只指出“视觉不够好、布局不够舒服、卡片太多”，但没有先判断信息是否完整、价值是否讲清、疑问是否解决，这个诊断不完整。

## 页面结构

serious public Tool 页面不要求固定模块顺序，但必须覆盖：

1. H1：具体工具任务。
2. Intro：一句到两句说明适合谁、输入什么、输出什么。
3. 工具工作台：输入区、操作按钮、结果/等待/错误状态。
4. 场景相关视觉物料或产品预览。
5. 你可以得到什么：列出用户完成操作后能拿到的具体结果、文件、报告、草案、清单、建议、链接、下载项或下一步。
6. 用户价值模块：说明这些结果能帮用户节省什么、避免什么、判断什么、交付什么或继续完成什么工作。
7. 使用说明或流程解释：让用户知道怎么开始。
8. 产品价值模块：核心能力、优势、差异化、结果质量、节省成本、可信度或适用场景。
9. 疑问消除模块：支持范围、失败原因、准确性、数据/隐私、导出、复核、限制或下一步。
10. Result guidance：如何理解、优化、导出或使用结果。
11. Use cases：真实场景，不堆关键词。
12. FAQ：只回答真实搜索问题。
13. Related tools 或下一步 CTA。
14. 更新时间、状态、能力边界或风险说明。

工具工作台必须是核心，不要把页面做成文章后面加表单。

模块形式可以自由选择，例如对比、流程、案例、结果拆解、能力卡片、风险说明、格式支持、失败原因、trust boundary、use case、FAQ、下一步 CTA。判断标准不是“有没有某个固定模块”，而是是否充分展示产品价值并回答用户疑问。

## 搜索需求覆盖图

每个 serious public Tool 页面都必须覆盖一张隐性的“搜索需求图”，不是只围绕主关键词写几段话。用户会用不同语言搜索同一个任务，也会搜索失败原因、替代方案、格式、边界和下一步。

页面正文应自然覆盖这些需求类型：

- **核心任务词**：工具名、主要动作和主要对象，例如下载、生成、分析、转换、检查、压缩、提取。
- **同义和平台变体**：用户可能使用的新旧名称、缩写、英文/中文混写、产品别名、对象别名。
- **输入/输出词**：支持的 URL、文件、文本、图片、格式、地区、语言、导出类型、报告/草案/清单/文件。
- **失败和排查词**：为什么失败、为什么识别不了、为什么没有结果、超时怎么办、链接/文件/格式不支持怎么办。
- **边界和权限词**：免费、登录、隐私、版权、商用、公开/私密、平台规则、人工复核、数据处理。
- **结果使用词**：生成后怎么用、怎么复制、下载、导出、发布、复核、继续编辑、进入下一步工具。
- **替代方案词**：手工、模板、脚本、插件、竞品、传统流程，以及 Tomako 为什么更适合。

不要把这些词机械堆进 FAQ。更好的做法是把搜索需求分散到 H2、段落、列表、结果指导、失败排查、支持范围和相关工具里，让页面像一个真正有帮助的工具页。

如果页面只覆盖主关键词，没有覆盖长尾、失败、边界、结果使用和下一步，视为内容不完整。

## 文案规则

必须做到：

- H1、title、description、intro、widget 讲同一个任务。
- 第一屏让用户判断“这是不是我要找的工具”。
- 每个 H2 对应一个真实问题、步骤、异议、判断或下一步。
- 先给有用答案，再补背景解释。
- 使用具体名词和动词。
- 使用用户语言，不使用内部团队语言。
- 把产品经理、设计师、研发的内部笔记转成用户能理解的任务语言。
- CTA 描述真实动作，例如“生成草案”“分析 URL”“下载清单”。
- 段落短，列表只用于真实集合。
- 表格只用于真实比较、结构化选项或结果解释。
- 工具名称、按钮、状态和说明使用用户动作语言。下载器就写“获取下载链接 / 下载文件”，转换器就写“转换格式”，分析器就写“生成报告”；不要把实现动作写成用户任务。
- 首屏副标题必须说明用户输入、工具动作和输出结果，不要用内部技术结论代替用户价值。
- 页面中的 metadata 卡、状态卡和标签必须帮助用户判断能力、范围、格式、时效或限制；不要展示对用户无意义的内部分类。

禁止：

- “提升效率”“释放增长”“AI 魔法”“一键优化全部”这类无具体输出的泛话。
- 在公开页面泄露 `LLM Task`、`Skills-OL`、`cc-connect`、prompt version、schema name、完成百分比等内部术语，除非目标用户就是开发/运营人员并且需要知道。
- 在普通用户页面把“Agent 方案评估”“无需 Agent”“是否使用 LLM”“前端/后端技术取舍”作为核心公开模块。它们属于内部实现或发布风险，默认写到开发记录，不写到 SEO 页面主体。
- metadata 或 Hero 承诺运行时做不到的能力。
- FAQ 每题都堆关键词。
- 用固定字数、关键词密度代替真实可用性。

## 公开模块必须服务用户任务

每个公开 section 都要回答用户真实问题。若一个 section 换成其它工具名称也几乎成立，说明它太泛。

每个 section 在写入前必须标注主要目的：

- **价值型**：展示能力、优势、差异化、结果质量、节省时间、专业性或适用场景。
- **疑问型**：回答用户顾虑、边界、失败原因、数据处理、准确性、导出、复核、版权/隐私等问题。
- **操作型**：帮助用户完成输入、生成、下载、复制、复核或下一步。

如果一个 section 不属于这三类之一，默认删除或重写。

必须避免：

- 通用“流程卡”只写输入、处理、输出，却没有体现当前工具对象。
- 用内部方案评估替代结果指导。
- 用“这个工具不用 AI/Agent”作为主要卖点，除非用户搜索意图确实关心离线、本地、隐私或成本。
- 把技术限制写成用户看不懂的后端描述。
- 只写功能列表，但没有说明为什么它对用户有价值。
- 只写漂亮卖点，但没有回答用户会担心的问题。
- 用 FAQ 承担所有解释，导致主体模块没有产品价值感。

## 必备价值模块：你可以得到什么

每个 serious public Tool 页面都必须有一个清楚的“你可以得到什么 / 用户会获得什么”模块。它不是 metadata，也不是阅读时间、工具分类或关键词标签，而是从用户视角列出操作完成后的实际收获。

这个模块必须回答：

- 用户提交输入后能拿到什么具体东西。
- 这些东西有什么用。
- 用户下一步可以拿它去做什么。
- 如果失败，用户能得到什么可操作反馈。

写法要求：

- 用具体名词，不用抽象收益。写“GIF 动图文件、MP4 源文件、失败原因提示、可复制的来源链接”，不要只写“更高效率”。
- 每一项最好是“结果 + 价值”，例如“可下载 MP4 文件：方便剪辑或保存素材”。
- 至少包含 3 到 5 个高价值结果项。
- 对生成/分析类工具，结果项应覆盖草案、报告、评分、建议、风险、缺失信息、复核清单、导出项等。
- 对下载/转换/提取类工具，结果项应覆盖文件、格式、预览、下载/复制动作、失败原因、边界提示等。
- 对检查/合规/安全类工具，结果项应覆盖问题列表、严重程度、修复建议、风险说明、复查路径等。

禁止：

- 用“阅读时间”“工具分类”“更新时间”“热门标签”占据这个模块的位置。
- 只写关键词 tag，例如“Twitter GIF / X GIF 下载 / GIF 转换”，但不说明用户能获得什么。
- 把这个模块写成内部实现过程。
- 把这个模块只塞进 FAQ 或页面底部。它应该出现在用户做决定的位置，例如工具工作台附近、工具工作台下方、首段说明之后或结果区之前。

更好的替代方向：

- 下载/转换/提取类：支持哪些链接或文件、输出格式、保存方式、失败原因、版权/隐私边界、结果可否再次编辑或发布。
- 生成类：需要什么输入、输出有哪些部分、缺失信息如何补、怎样复核、怎样继续修改、适合哪些发布/工作流场景。
- 分析/报告类：数据来源、报告结构、可信度、缺失信息、结论如何使用、下一步如何优化。
- 检查/清单类：检查项目、严重程度、失败/不通过原因、修复建议、复查方式和导出方式。

## 必备解释模块

除非工具极其简单，否则页面主体中要优先考虑这些解释模块。模块名称可以变化，但内容不能缺位：

- **支持范围**：当前工具支持什么输入、对象、格式、语言、地区、平台、结果类型。
- **不支持范围**：哪些输入、对象、格式、私密内容、权限状态、复杂场景或风险类型不支持。
- **失败排查**：当用户拿不到结果、结果为空、链接/文件无效、格式不对、服务超时或第三方限制时，应该检查什么、怎么恢复。
- **结果使用指导**：用户拿到结果后如何复制、下载、导出、复核、发布、继续修改或进入下一步工具。
- **相关工具和下一步**：把页面接入工具集群，让用户能继续完成同一工作流。

这些模块不一定都用独立 H2，但必须在页面中以可抓取 HTML 文本出现。FAQ 可以补充，但不能替代主体解释。

按工具类型补充：

- 下载/抓取/转换/提取：必须说明支持范围、不支持范围、失败原因、权限/版权/隐私边界、输出格式和保存方式。
- 生成/写作/命名/素材：必须说明输入要求、输出结构、如何挑选/复核/修改、结果适用场景和不可承诺内容。
- 分析/评分/报告：必须说明数据来源、结论含义、可信边界、缺失信息、下一步操作和导出方式。
- 检查/合规/安全/法务：必须说明检查范围、风险等级、人工复核、非专业意见边界、修复路径和再次检查方式。

## 产品价值表达

价值表达必须具体，不做空泛营销。

## 超级 P0：必须讲清楚为什么选我们

公开 SEO Tool 页面不能只说明“它怎么工作”，必须把产品价值讲出去。用户不关心内部流程本身，用户关心用了以后能得到什么、为什么这个结果值得信任、为什么不用别的工具或自己手动做。

每个 serious public Tool 页面必须有清晰的价值主张，并在页面主体中回答：

- 为什么用 Tomako 这个工具。
- 它比手工、模板、脚本、浏览器插件、普通文章或竞品更适合当前任务的地方。
- 什么样的同类工具才算好。
- Tomako 如何满足这些好工具标准。
- 用户最终得到的结果能帮他省什么、避免什么、提升什么。

“好工具标准”必须从用户真正关心的决策标准出发，而不是从产品内部流程出发。优先级通常是：

- **稳定性**：能不能稳定完成任务，失败率是否可控，失败时是否可恢复。
- **速度**：等待时间是否合理，流程是否少步骤。
- **结果质量**：文件/草案/报告/建议是否清晰、可用、可复核，质量是否尽量保真。
- **成本**：是否免费、低成本、少登录、少配置，是否减少人工或专业门槛。
- **易用性**：输入是否简单，结果是否能直接复制、下载、导出、继续编辑。
- **兼容性**：是否支持用户常见格式、平台、语言、地区、设备或工作流。
- **可信与安全**：隐私、版权、合规、数据处理、人工复核或平台边界是否清楚。

过程透明、状态说明、失败原因提示、边界解释是重要补充，但不能冒充主要标准。不要把“用户在输入前知道什么”“处理时知道发生了什么”这类过程信息写成最核心的好标准；它们应该服务稳定、速度、质量、易用或可信这些用户价值。

不同工具类型可以这样组织“好标准”：

- 下载/转换类：下载稳定、速度快、尽量保留原始质量、输出格式可用、免费或低成本、操作少、失败可恢复、版权/隐私边界清楚。
- 生成类：输出质量高、结构完整、贴合输入、可编辑、可导出、生成速度合理、缺失信息和风险可复核。
- 分析类：结论准确可用、来源清楚、严重程度有区分、建议可执行、能导出或交给下一步处理。
- 检查类：覆盖范围清楚、问题识别稳定、误报少、修复建议具体、支持复查、风险等级可信。

允许在需求不完整时做产品化假设和价值包装，但必须遵守：

- 可以把真实功能包装得更清楚、更有吸引力。
- 可以基于当前页面能力推导用户收益，例如少步骤、结构化结果、边界更清楚、失败提示更可读。
- 不得编造不可验证事实，例如官方合作、行业第一、最快、100% 成功、真实客户数据、第三方认证、法律保证、平台背书或具体性能数字。
- 不得承诺 runtime、后端、下载、抓取、生成质量或审核能力没有实现的东西。

“它是怎么工作的”不是必选模块。若保留，必须改写成用户结果导向，例如：

- “你会得到什么”
- “为什么这个结果更容易使用”
- “一个好的 X 工具应该具备什么”
- “使用前需要知道的边界”
- “下载/生成/分析失败时怎么办”

禁止把内部实现写给普通用户看，例如 FFmpeg、Agent 方案、LLM Task、Skills-OL、后端取舍、是否使用 Agent 等。除非目标用户明确是开发者，否则这些内容只属于开发记录。

可以从这些角度组织模块：

- 核心能力：工具真正完成了什么关键动作。
- 结果质量：输出为何可用、可复核、可导出、可继续修改。
- 差异化：比模板、手工、脚本、普通文章或竞品更好的地方。
- 成本收益：节省时间、减少重复劳动、降低试错、减少专业门槛。
- 专业感：结构化输出、风险提示、来源说明、格式支持、边界清楚。
- 工作流衔接：用户拿到结果后能复制、下载、发布、复核、继续编辑或进入下一个工具。

价值模块必须落到真实功能和输出，不要写成抽象品牌口号。

价值表达不能只出现在 FAQ 或页面底部。首屏 intro、工具工作台附近或主体前半部分至少要出现一次明确的“为什么值得用”。

## 用户疑问地图

页面必须主动猜想用户疑问，并在合适位置回答，不一定都放 FAQ。

常见疑问类型：

- 输入疑问：我需要提供什么？支持什么格式、链接、语言、地区？
- 输出疑问：会得到什么？能复制、下载、导出、继续编辑吗？
- 准确性疑问：结果可靠吗？什么时候会失败？失败后怎么办？
- 边界疑问：哪些内容不支持？哪些结果需要人工复核？
- 隐私疑问：提交的 URL、文本、文件会怎样处理？
- 成本疑问：是否免费、是否需要登录、是否要等待？
- 权限疑问：第三方平台、版权、商用、私密内容是否允许？
- 下一步疑问：拿到结果后应该怎么用，如何优化，如何进入后续流程？

疑问回答要放在用户需要它的位置：输入前回答“能不能用”，结果前回答“等什么”，结果后回答“怎么用”，页面底部再用 FAQ 收尾。

特别注意：失败排查和结果使用指导通常比普通 FAQ 更有 SEO 价值。不要只写“为什么失败？”这种一问一答，要让用户能照着页面完成下一步。

## 可抓取内容

重要 SEO 内容必须是 HTML 文本：

- H1。
- intro。
- 关键说明。
- 使用步骤。
- 输出解释。
- FAQ。
- 能力边界。
- 相关链接。

不要只把核心信息放在图片、canvas、客户端交互后隐藏状态或生成结果里。图片用于提高质感，不替代正文。

## Metadata

公开页面必须定义或继承：

- 本地化 title。
- 本地化 description。
- canonical。
- hreflang alternates。
- Open Graph。
- robots/index 决策。
- sitemap eligibility。
- 稳定且有价值时补 JSON-LD。

规则：

- title 包含具体任务和主要结果，不写空泛“免费工具”。
- description 说明输入、动作、输出和边界，像搜索结果里值得点击的一句话。
- metadata 不得承诺页面或运行时没有的能力。
- alt 描述图片有用主体，不堆关键词。
- `ToolSpec` 中的 locales 主要用于 SEO/list metadata，正文放 i18n message。

## FAQ

只有真实有搜索价值或用户决策价值时才加 FAQ。

常见好问题：

- 需要提供什么输入？
- 输出准确性如何？
- 能否复制、下载、商用？
- 提交的 URL 或数据会怎样处理？
- 这是 AI 生成还是确定性计算？
- 拿到结果后下一步做什么？
- 结果是否需要人工复核？
- 对第三方平台、下载、抓取或转换类工具：支持哪些链接/格式，为什么会失败，是否保存用户输入，是否能处理私密内容，用户使用下载内容需要注意什么。

避免：

- 为了 SEO 强行写 FAQ。
- 问题重复关键词。
- 回答承诺未上线集成。
- 把 FAQ 写成长篇百科，埋掉工具本体。
- 用内部问题替代用户问题，例如“这个工具用到了 Agent 吗？”“为什么不用 LLM？”除非目标用户明确是开发者或隐私敏感用户。
- 把本应在主体中解释的核心价值全部塞进 FAQ。FAQ 是补充答疑，不是产品价值表达的主阵地。

## 多语言

本地化不是直译。

必须：

- 每个 locale 重新确认关键词和自然表达。
- 使用当地用户熟悉的产品分类、法规/地区词、例子和语气。
- H1、metadata、intro、FAQ、widget labels 在每个 locale 内一致。
- 避免保留英文习语、双关或内部缩写。
- 检查 `hreflang`、canonical、sitemap、Open Graph locale、URL 与文案匹配。

## 能力边界

当页面涉及 mock、demo、AI、抓取或专业领域，必须说明边界：

- 本地模板还是云端 Agent。
- 是否为草案。
- 是否需要人工/律师/财务/安全专业复核。
- URL 抓取可能受登录、robots、反爬或动态渲染影响。
- 输出不是保证、排名、法律意见、财务意见或安全审计结论。
- 第三方平台内容下载、解析或转换工具必须说明：只处理用户有权访问或使用的内容；私密、删除、受限、登录后内容可能失败；平台规则、版权和隐私责任由用户确认。

Agent-backed 工具的文案必须和真实 runtime 一致。脚本只能做确定性检查时，不要写成个性化专家建议。

## 内部链接

每个工具页应有：

- tools index 可达入口。
- related tools 或下一步 CTA。
- MCP guide link，当 MCP 启用且对用户有意义。
- 模板、博客、帮助文档链接，仅在帮助同一任务时加入。

页面集群按用户工作流互链，不按关键词机械互链。

相关工具应该服务用户下一步，而不是泛泛返回首页。例如下载类工具可链接到格式转换、压缩、素材整理；生成类工具可链接到改写、检查、导出；分析类工具可链接到修复、复查、报告生成。

## AI Search / GEO

把 GEO 当作 SEO 基本功：

- 顶部给出清晰答案和输出承诺。
- heading 层级清楚。
- section 短而自包含。
- 必要时使用例子、表格或列表帮助提取。
- 事实性、指导性内容优先引用权威来源。
- 不加隐藏文案或“LLM-friendly”废话块。
