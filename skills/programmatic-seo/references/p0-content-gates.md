# P0 文案与 SEO 内容门槛

## 使用时机

涉及 SEO 文案、H1、metadata、FAQ、页面模块、可搜索性、可抓取内容、能力边界、内部链接、多语言文案，或对已有 Tool 页面 / 内容页做关键词与用户需求驱动的文案校准时，必须读取本文件。

本文件合并了旧设计规范中的 SEO 与文案要求，并吸收外部 SEO playbook 的可用检查点。外部资料只能作为审查视角，不能覆盖 Tomako 架构和真实运行能力。

## 适用范围

本文件是共享内容门槛，适用于：

- 交互型 Tool Page：还必须继续读取 `p0-ui-gates.md`、`p0-asset-gates.md`、`p0-runtime-gates.md` 和 `tool-page-implementation.md`。
- 内容型 SEO Page：还必须继续读取 `p0-content-page-brief.md`、`p0-content-page-structure-gates.md`、`p0-content-page-writing-gates.md` 和 `content-page-implementation.md`。

判断标准：先确认用户搜索需求和页面价值，再决定它应该做成交互工具、内容页、模板页，还是内容集群的一部分。

## SEO 页面的内容目标

程序化 SEO 不是批量换关键词，而是用可复用系统生产真正有用、可抓取、可索引、能解决用户任务的页面。

SEO 页面的模块可以自由组织，不需要固定套用某几个 section。但除工具交互、正文解释和基础使用说明外，页面主体必须服务两个核心目标：

1. **体现产品价值**：把功能讲得足够漂亮、具体、可信，突出核心能力、优势、差异化和用户拿到的好处。
2. **解决用户疑问**：预判用户使用前、使用中、使用后的疑虑，并用清楚、低成本、让人放心的方式回答。

每个 SEO 页面必须回答：

- 用户为什么搜索这个页面。
- 用户想完成什么任务或做什么判断。
- 如果是 Tool 页：用户要输入什么，工具会做什么。
- 如果是内容页：读完后用户会知道什么、拿到什么、决定什么或进入哪个下一步。
- 用户会得到什么。
- 结果应该怎么使用。
- 页面有哪些限制。
- 这个页面为什么值得读或值得用，而不是随便看一篇普通文章、模板、脚本或竞品页面。
- Tomako 相比替代方案更好在哪里，用户为什么应该继续使用 Tomako 的工具、模板或工作流。
- 用户可能担心什么，页面如何让他放心继续操作。

## 公开信息与内部信息分级

写 SEO 页面文案、ToolSpec、i18n、FAQ、metadata、alt、截图说明或页面结构前，必须先把信息分成“可公开给用户看”和“只用于实现或内部协作”。不能把后端校验值、前端常量、内部 prompt 或团队备注直接转成页面正文。

分级规则：

- **公开用户文案**：用户做决定必须知道的信息，例如工具能做什么、需要什么输入、会得到什么结果、支持/不支持什么、失败怎么办、数据如何处理、是否需要人工复核。
- **交互状态文案**：用户在特定动作后才看到的信息，例如弹窗标题、输入提示、错误提示、下载成功提示。它可以解释用户下一步动作，但不能泄露验证码、邀请码、密钥或内部判断依据。
- **前端代码信息**：默认视为可被用户查看。浏览器包、HTML、JSON、静态资源、ToolSpec catalog、i18n 文案、source map、客户端常量、`NEXT_PUBLIC_*` 环境变量都不能放邀请码明文、API key、私有下载地址、内部账号、prompt、策略版本或隐藏审核备注。
- **后端 / env / 内部记录**：用于校验、鉴权、调用 provider、记录发布风险或团队协作的信息，例如邀请码真实值、API key、账号 token、provider 响应、内部 prompt、schema name、Skills-OL/cc-connect 路由、成本、灰度策略、审核备注。默认不进入公开页面、FAQ、metadata、JSON-LD、图片文字或前端 bundle。

写法边界：

- 可以公开写“下载需要邀请码”“请在弹窗中输入邀请码”“仅供内部 Beta 测试”。
- 不可以公开写“当前邀请码是 123456”“前端校验码为 xxx”“后端接口地址是 xxx”“prompt version 是 xxx”。
- 如果页面需要说明边界，只说明用户能理解的事实：需要权限、需要邀请码、暂不开放、需要人工复核、结果仅供参考。
- 如果某个值必须用于校验，优先放到后端或环境变量；前端只提交用户输入并接收通过/失败，不在页面正文或客户端常量里保存真实值。
- 如果当前版本暂时只能前端校验，公开文案仍不能写出真实值；必须在交付风险中说明“前端校验不等于真正保密”，并推动后端/env 校验。

落地检查：

- 搜索文案和配置中是否出现 `当前邀请码`、`校验码`、`API key`、`token`、`secret`、`prompt version`、`Skills-OL`、`cc-connect`、`schema`、内部账号、内部下载地址或具体数字口令。
- 检查 `ToolSpec`、`src/generated/tools-spec-catalog.json`、i18n messages、FAQ、metadata、alt、截图素材和页面底部说明，不只检查正文。
- 发现不可公开信息时，不是简单换个位置，而是改成公开边界表达；真实值移到后端/env/内部记录。

## 关键词证据流程

写文案或校准已有页面文案前，必须先确定关键词和用户需求，再写文案或改文案，再进入页面生成或页面更新。关键词不是越多越好，先用上下文收敛，再用数据验证。

强制顺序：

1. **需求摘要**：根据用户输入总结工具名、目标用户、任务目标、用户输入、用户最终期望结果、适用场景、目标语言和地区，并返回到对话框。
2. **关键词候选小表**：基于上下文生成候选词，返回到对话框。表格至少包含候选词、搜索意图、为什么可能有人搜、与工具匹配度、风险、是否进入第一轮查询。
3. **优先复用 Brief**：先检查 `Tomako/docs/seo-keyword-briefs/{slug}.md` 和 `.json`。如果已有且工具范围、目标市场和时间没有明显变化，直接复用，不重复查询付费 API。
4. **数据验证**：如果没有可复用 brief、DataForSEO 可用且上下文足够，在 `Tomako/` 运行 `pnpm seo:keyword-brief -- --slug <tool-slug> --keyword "primary keyword"`。查询多少关键词由实际场景决定，不设固定默认数量；CLI 单次硬上限是 50 个关键词。需求清楚、候选词质量高时可以多查；需求模糊、范围不清或能力未确认时先问问题或先做假设，不调用付费 API。不要只依赖 slug 自动推词。
5. **小步扩展**：根据搜索量、CPC、付费竞争度、SERP Top 10 和意图判断，再决定是否扩展。不要为了覆盖而批量铺量；超过 CLI 单次上限时，应拆分主题、分批查询或先缩小范围。
6. **Keyword Evidence Brief**：把最终给页面文案使用的信息返回到对话框，再进入文案写作。

DataForSEO 是证据工具，不是无脑铺量工具。标准入口是 Tomako repo 内的本地 CLI 和 `docs/seo-keyword-briefs/` 缓存，不再要求跨到 `lark-product-monitor`。没有足够上下文时，先问问题或写明假设，不要在需求模糊时批量查询。若没有 API key、目标市场不明确、关键词过泛或工具能力还没确认，先做 AI 关键词假设，不调用付费 API。

这套流程同时适用于：

- **0 到 1 新页面**：先产出需求摘要、候选关键词、Keyword Evidence Brief、Copy Brief 和信息模块清单，再实现页面。
- **已有页面文案校准**：先读取当前页面文案、ToolSpec、i18n 和已有 brief，再对照关键词证据诊断主关键词是否准确、H1/title/description 是否匹配、用户价值是否讲清、FAQ 是否覆盖真实疑问、是否存在薄内容或无意义模块，最后再改文案和必要模块。

标准命令：

```bash
pnpm seo:keyword-brief -- --slug <tool-slug> --keyword "primary keyword"
```

如果已经有可复用 brief，直接读取 brief，不要刷新。若需要刷新，必须说明刷新原因，例如工具范围变化、目标市场变化、SERP 明显变化或旧 brief 只是 scaffold。

没有 DataForSEO 密钥时，标准命令不会写入可提交 brief。只有明确需要低置信占位时，才允许使用 `--scaffold`，并且必须在对话框说明它不是数据证据，不能当成最终 Keyword Evidence Brief。

可提交到 Git 的内容：

- `docs/seo-keyword-briefs/{tool-slug}.md`
- `docs/seo-keyword-briefs/{tool-slug}.json`

不能提交到 Git 的内容：

- `.env`、`DATAFORSEO_LOGIN`、`DATAFORSEO_PASSWORD` 或部署密钥。
- 不想公开的原始 provider 响应。需要保留原始响应时放到本地 ignored 目录 `docs/seo-keyword-briefs/raw/`。
- 无密钥生成的 scaffold brief 默认不要提交为正式数据依据；除非明确标注低置信、仅用于临时占位。

第一轮种子词选择标准：

- 最像用户真实会搜索的任务表达。
- 与工具真实输入、动作、输出强相关。
- 有明确工具意图或可转化意图。
- 能代表一组相似需求。
- 不选择过泛、百科化、品牌无关或明显不承接当前工具的词。

DataForSEO 结果判断：

- 搜索量判断需求大小。
- CPC 和付费竞争度判断商业价值和竞争压力。
- SERP Top 10 判断搜索意图：工具页、模板页、教程页、官方文档、竞品页或聚合页。
- 如果 SERP 和工具能力不匹配，不要硬追高搜索量词。

Keyword Evidence Brief 格式：

```text
需求摘要：
主关键词：
选择理由：
搜索量 / CPC / 付费竞争度：
SERP 意图：
次关键词：
不建议追的词：
页面必须覆盖的用户问题：
建议 H1：
建议 title：
建议 description：
建议 H2 / 模块：
FAQ 关键词来源：
证据来源：AI 推断 / DataForSEO / SERP / Search Console / 竞品 / 用户提供
查询轮次和查询词：
置信度：
仍需验证：
```

对话框必须展示：

- 第一步需求摘要。
- 第二步关键词候选小表。
- 最终 Keyword Evidence Brief。
- 若是已有页面文案校准，还必须展示当前页面的文案缺口：关键词缺口、用户价值缺口、疑问覆盖缺口、需要删除或降级的无效模块。

这些输出不要求用户逐步确认，但必须可见，方便团队校准关键词假设、查询成本和文案方向。

## 内容先行与信息模块清单

新建、重做、优化或诊断 SEO 页面时，必须先完成内容判断，再进入 UI 或 MDX 实现。页面不是先选排版模板再填文案，而是先从用户视角和 SEO 增长视角判断“这个页面必须回答什么、证明什么、消除什么疑虑”。

UI 设计前必须先列出信息模块清单。格式可以简短，但必须覆盖：

```text
模块名称：
用户问题：
SEO / 搜索意图：
核心信息：
产品价值或用户疑虑：
期望用户动作：
适合位置：
UI 呈现建议：
```

执行顺序：

1. 先判断用户搜索这个工具时真正想完成的任务。
2. 再完成关键词候选小表和 Keyword Evidence Brief。
3. 再列出必须出现的信息：输入、输出、可获得结果、价值、为什么选我们、好工具标准、支持范围、不支持范围、失败排查、结果使用、相关工具。
4. 再判断每个信息模块属于价值型、疑问型还是操作型。
5. 再安排模块优先级和页面位置。
6. 最后才进入 UI、配图、布局和组件选择。

P0 禁止：

- 先画 Hero、卡片、两栏、FAQ、步骤图，再回头给这些容器补文案。
- 因为某个 UI 模板好看，就保留没有用户价值或搜索价值的模块。
- 把 UI 排版问题误当成文案问题，只调样式不补用户价值、搜索需求或疑问解答。
- 让视觉图、metadata 卡或流程卡替代“你可以得到什么”“为什么选我们”“支持与失败边界”等核心信息。

如果当前页面诊断只指出“视觉不够好、布局不够舒服、卡片太多”，但没有先判断信息是否完整、价值是否讲清、疑问是否解决，这个诊断不完整。

## Tool 页面结构

serious public Tool 页面默认按“看见工具 -> 明白价值 -> 降低门槛 -> 建立信任 -> 消除顾虑 -> 推动行动”的说服漏斗组织。可以根据工具类型微调顺序，但不能省略核心作用；如果改动顺序，必须能说明用户决策路径更清楚。

默认骨架自上而下：

1. H1 + 首屏摘要：H1 说清核心关键词、动作和结果；副标题补充痛点、动作和结果；完整输入、用途和边界下移到正文或工作台附近。
2. 工具工作台：输入区、操作按钮、结果/等待/错误状态必须靠近顶部，让用户立刻看见工具本体。
3. 你会得到什么：用 bullet、结果预览或示例图展示输出长什么样，这是核心转化区。
4. 三步以内使用流程：让用户知道如何开始，步骤文案使用动作词和场景词。
5. 为什么值得用：说明 Tomako 相比手工、模板、脚本、插件或竞品的真实用户收益。
6. 具体应用场景：覆盖 3 到 5 个典型场景，让用户对号入座，并自然布局长尾场景词。
7. 支持范围 / 不支持范围：明确输入格式、对象、权限、输出类型、复核要求和暂不支持内容。
8. 结果后续操作指南：说明用户拿到结果后如何复制、下载、导出、复核、发布、交给团队或进入下一步工具。
9. FAQ：收尾解决真实疑问；每页至少 2 条问题必须是当前工具独有问题，不能换个工具名也成立。
10. 相关工具 / 下一步 CTA：用内部链接承接同一工作流。

除默认骨架外，页面仍需按工具风险补充场景相关视觉物料或产品预览、更新时间、状态、能力边界或风险说明。

Hero 文案长度限制：

| 模块 | 核心作用 | SEO 要点 | 长度限制 |
| --- | --- | --- | --- |
| H1 + 首屏摘要 | 一屏内说清页面任务和核心价值 | H1 嵌入主关键词，副标题自然承接长尾意图 | H1 <= 15 字；副标题 <= 30 字；PC 端 Hero 区总计不超过 3 行 |

超过长度的痛点、输入、用途、边界或补充说明，应拆到工作台帮助、正文首段、“你会得到什么”或支持范围中，不要塞进 Hero。

工具工作台必须是核心，不要把页面做成文章后面加表单。

模块形式可以自由选择，例如对比、流程、案例、结果拆解、能力卡片、风险说明、格式支持、失败原因、trust boundary、use case、FAQ、下一步 CTA。判断标准不是“有没有某个固定模块”，而是是否充分展示产品价值并回答用户疑问。

## 内容页结构

内容型 SEO 页面不要强行包含工具工作台或结果区。它必须按搜索意图选择结构，并继续满足“体现产品价值”和“解决用户疑问”两个目标。

内容页必须覆盖：

1. H1：匹配用户搜索问题或任务。
2. Intro：说明适合谁、读完能得到什么、边界是什么。
3. 直接答案或路线：尽早给用户可用结论。
4. 主体内容：按教程、模板、对比、解释、清单、排查、hub 等页面类型展开。
5. 示例、模板、判断标准、步骤、表格或可复制材料。
6. 风险边界、常见误区或失败排查。
7. 与 Tomako 工具、模板或工作流的关系。
8. FAQ：覆盖真实疑问。
9. 相关内容和下一步 CTA。

内容页的详细结构、写作和工程落地见 `p0-content-page-brief.md`、`p0-content-page-structure-gates.md`、`p0-content-page-writing-gates.md` 和 `content-page-implementation.md`。

## 搜索需求覆盖图

每个 serious public SEO 页面都必须覆盖一张隐性的“搜索需求图”，不是只围绕主关键词写几段话。用户会用不同语言搜索同一个任务，也会搜索失败原因、替代方案、格式、边界和下一步。

每个页面至少覆盖以下 7 类搜索需求词，每类至少 1 个，并分散在自然语境中：

- **主关键词**：用户搜索的核心工具名或任务名，例如 AI 海报生成器、PDF 压缩工具。
- **同义词**：相同任务的不同叫法，例如海报制作工具、图片压缩器、文案改写器。
- **输入词**：用户用什么开始，例如产品 URL、截图、关键词、CSV、社媒链接。
- **输出词**：用户能拿到什么，例如官网首屏海报、Brief 文档、风险报告、下载链接。
- **结果使用词**：用户拿到结果后怎么用，例如发给设计、复制到广告后台、导出给团队复核。
- **边界词**：用户关心能不能，例如支持 JPG、是否能商用、是否需要人工复核、不支持 PSD。
- **替代方案词**：用户对比什么，例如手工做、模板、脚本、插件、Canva、竞品或传统流程。

失败和排查词也必须按工具风险补充，例如为什么失败、为什么识别不了、为什么没有结果、超时怎么办、链接/文件/格式不支持怎么办。它们可以归入边界词或 FAQ，但更优先放在工作台错误态、支持范围、失败排查和结果使用指导中。

关键词部署位置：

- H1：必须包含 1 个主关键词。
- H2：优先覆盖同义词、场景词或结果词，不要硬塞主关键词。
- H3：覆盖长尾场景词、平台词、用途词或人群词。
- 首段正文：自然出现主关键词，并补 1 到 2 个长尾词。
- Use Cases：承接长尾场景词。
- FAQ：承接疑问式长尾词。
- Meta Description：包含主关键词、输入/动作/输出和行动理由。

不要把这些词机械堆进 FAQ。更好的做法是把搜索需求分散到 H2、段落、列表、结果指导、失败排查、支持范围和相关工具里，让页面像一个真正有帮助的 SEO 页面。

如果页面只覆盖主关键词，没有覆盖长尾、失败、边界、结果使用和下一步，视为内容不完整。

## 标题体系

每个标题都必须回答一个真实用户问题，并自然覆盖对应层级的关键词。标题不是装饰，也不是关键词容器。

- **H1**：告诉用户和搜索引擎“这个页面是干什么的”。必须包含主关键词，只写“动作 + 结果”或“核心工具名 + 核心结果”，控制在 15 字以内；不要把痛点、输入、用途和边界都塞进 H1。
- **H2**：回答“这个工具为什么值得用”“怎么用”“会得到什么”或“要注意什么”。优先覆盖同义词、场景词或结果词，推荐用动词或利益词加用户场景/工具能力。
- **H3**：解释 H2 下的具体功能、平台、人群或用途。优先覆盖长尾场景词，例如平台名、用途、人群、格式、行业或工作流。

标题验收：

- H1、title、description、intro、widget 必须讲同一个任务。
- 每个 H2 都能转写成一个用户问题、步骤、异议、判断或下一步。
- H3 必须服务它所在的 H2，不得为了塞长尾词单独漂浮。
- H1 控制在 15 字以内，Hero 副标题控制在 30 字以内；PC 端 Hero 区总计不超过 3 行。
- 标题读起来必须像自然语言，不得重复堆主关键词。

## 文案规则

必须做到：

- H1、title、description、intro、widget 讲同一个任务。
- 第一屏让用户判断“这是不是我要找的工具”。
- 使用场景区分：主标题在包含主关键词的前提下，只表达“动作 + 结果”；副标题用“痛点 + 动作 + 结果”；正文段落才完整使用“痛点 -> 输入 -> 动作 -> 结果 -> 用途 -> 边界”六段式。
- 每个 H2 对应一个真实问题、步骤、异议、判断或下一步。
- 先给有用答案，再补背景解释。
- 使用具体名词和动词。
- 使用用户语言，不使用内部团队语言。
- 把产品经理、设计师、研发的内部笔记转成用户能理解的任务语言。
- CTA 描述真实动作，例如“生成草案”“分析 URL”“下载清单”。
- 段落短，列表只用于真实集合；超过 50 字的段落优先拆成列表或两段。
- 表格只用于真实比较、结构化选项或结果解释。
- 工具名称、按钮、状态和说明使用用户动作语言。下载器就写“获取下载链接 / 下载文件”，转换器就写“转换格式”，分析器就写“生成报告”；不要把实现动作写成用户任务。
- 首屏副标题必须说明用户输入、工具动作和输出结果，不要用内部技术结论代替用户价值。
- 页面中的 metadata 卡、状态卡和标签必须帮助用户判断能力、范围、格式、时效或限制；不要展示对用户无意义的内部分类。
- 语气像懂行的朋友在给建议：可信、专业、稳，少用夸张营销。
- 统一用“你”称呼用户，不写“用户可上传文件”这类说明书口吻。
- 句式以短句为主，优先使用“上传、生成、下载、查看、确认、复核、导出”等动作词。
- 主标题应控制在 15 字以内，副标题不超过 30 字。超过此长度的文案，应拆分为多行或分散到正文中。
- 问句可以增强代入感，但每个屏幕最多 1 到 2 个；感叹号必须克制，每页不超过 3 到 5 个。

正文价值主张六段式公式：

```text
[用户当前遇到的问题]，[输入什么]，[工具做什么]，[得到什么具体结果]，[结果怎么用]，[有什么边界/前提]。
```

完整六段式不要求全部进入 Hero。Hero 只承载主标题和副标题的短表达；完整输入、用途、边界和前提应分散到正文首段、工作台附近说明、结果说明或支持范围中，并在用户做决定前出现。

禁止：

- “提升效率”“释放增长”“AI 魔法”“一键优化全部”这类无具体输出的泛话。
- 在公开页面泄露 `LLM Task`、`Skills-OL`、`cc-connect`、prompt version、schema name、完成百分比等内部术语，除非目标用户就是开发/运营人员并且需要知道。
- 在公开页面、FAQ、metadata、ToolSpec、i18n 或前端常量中泄露邀请码明文、验证码、API key、token、私有下载地址、内部账号、后端校验值或团队审核备注。
- 在普通用户页面把“Agent 方案评估”“无需 Agent”“是否使用 LLM”“前端/后端技术取舍”作为核心公开模块。它们属于内部实现或发布风险，默认写到开发记录，不写到 SEO 页面主体。
- metadata 或 Hero 承诺运行时做不到的能力。
- FAQ 每题都堆关键词。
- 用固定字数、关键词密度代替真实可用性。

## 公开模块必须服务用户任务

每个公开 section 都要回答用户真实问题。若一个 section 换成其它工具名称也几乎成立，说明它太泛。

每个 section 在写入前必须标注主要目的：

- **价值型**：展示能力、优势、差异化、结果质量、节省时间、专业性或适用场景。
- **疑问型**：回答用户顾虑、边界、失败原因、数据处理、准确性、导出、复核、版权/隐私等问题。
- **操作型**：帮助用户完成输入、生成、下载、复制、复核或下一步。

如果一个 section 不属于这三类之一，默认删除或重写。

必须避免：

- 通用“流程卡”只写输入、处理、输出，却没有体现当前工具对象。
- 用内部方案评估替代结果指导。
- 用“这个工具不用 AI/Agent”作为主要卖点，除非用户搜索意图确实关心离线、本地、隐私或成本。
- 把技术限制写成用户看不懂的后端描述。
- 只写功能列表，但没有说明为什么它对用户有价值。
- 只写漂亮卖点，但没有回答用户会担心的问题。
- 用 FAQ 承担所有解释，导致主体模块没有产品价值感。

## 必备价值模块：你可以得到什么

每个 serious public Tool 页面都必须有一个清楚的“你可以得到什么 / 用户会获得什么”模块。它不是 metadata，也不是阅读时间、工具分类或关键词标签，而是从用户视角列出操作完成后的实际收获。

这个模块必须回答：

- 用户提交输入后能拿到什么具体东西。
- 这些东西有什么用。
- 用户下一步可以拿它去做什么。
- 如果失败，用户能得到什么可操作反馈。

写法要求：

- 用具体名词，不用抽象收益。写“GIF 动图文件、MP4 源文件、失败原因提示、可复制的来源链接”，不要只写“更高效率”。
- 每一项最好是“结果 + 价值”，例如“可下载 MP4 文件：方便剪辑或保存素材”。
- 至少包含 3 到 5 个高价值结果项。
- 对生成/分析类工具，结果项应覆盖草案、报告、评分、建议、风险、缺失信息、复核清单、导出项等。
- 对下载/转换/提取类工具，结果项应覆盖文件、格式、预览、下载/复制动作、失败原因、边界提示等。
- 对检查/合规/安全类工具，结果项应覆盖问题列表、严重程度、修复建议、风险说明、复查路径等。

禁止：

- 用“阅读时间”“工具分类”“更新时间”“热门标签”占据这个模块的位置。
- 只写关键词 tag，例如“Twitter GIF / X GIF 下载 / GIF 转换”，但不说明用户能获得什么。
- 把这个模块写成内部实现过程。
- 把这个模块只塞进 FAQ 或页面底部。它应该出现在用户做决定的位置，例如工具工作台附近、工具工作台下方、首段说明之后或结果区之前。

更好的替代方向：

- 下载/转换/提取类：支持哪些链接或文件、输出格式、保存方式、失败原因、版权/隐私边界、结果可否再次编辑或发布。
- 生成类：需要什么输入、输出有哪些部分、缺失信息如何补、怎样复核、怎样继续修改、适合哪些发布/工作流场景。
- 分析/报告类：数据来源、报告结构、可信度、缺失信息、结论如何使用、下一步如何优化。
- 检查/清单类：检查项目、严重程度、失败/不通过原因、修复建议、复查方式和导出方式。

## 必备解释模块

除非工具极其简单，否则页面主体中要优先考虑这些解释模块。模块名称可以变化，但内容不能缺位：

- **支持范围**：当前工具支持什么输入、对象、格式、语言、地区、平台、结果类型。
- **不支持范围**：哪些输入、对象、格式、私密内容、权限状态、复杂场景或风险类型不支持。
- **失败排查**：当用户拿不到结果、结果为空、链接/文件无效、格式不对、服务超时或第三方限制时，应该检查什么、怎么恢复。
- **结果使用指导**：用户拿到结果后如何复制、下载、导出、复核、发布、继续修改或进入下一步工具。
- **相关工具和下一步**：把页面接入工具集群，让用户能继续完成同一工作流。

这些模块不一定都用独立 H2，但必须在页面中以可抓取 HTML 文本出现。FAQ 可以补充，但不能替代主体解释。

工作台附近必须用轻量方式分散消除关键疑问，例如小字号说明、折叠帮助、图标列表、错误态提示或结果区 guidance。至少覆盖：

- 支持什么输入格式、链接、文件、语言、地区或对象。
- 暂不支持什么，例如私密内容、特定导出格式、实时编辑、专业终稿或平台受限内容。
- 生成失败或结果为空时用户应该怎么处理。
- 用户提交的数据、URL、截图或文本如何被使用。
- 结果能否商用、是否需要人工复核、是否能直接交付。

按工具类型补充：

- 下载/抓取/转换/提取：必须说明支持范围、不支持范围、失败原因、权限/版权/隐私边界、输出格式和保存方式。
- 生成/写作/命名/素材：必须说明输入要求、输出结构、如何挑选/复核/修改、结果适用场景和不可承诺内容。
- 分析/评分/报告：必须说明数据来源、结论含义、可信边界、缺失信息、下一步操作和导出方式。
- 检查/合规/安全/法务：必须说明检查范围、风险等级、人工复核、非专业意见边界、修复路径和再次检查方式。

## 产品价值表达

价值表达必须具体，不做空泛营销。

## 超级 P0：必须讲清楚为什么选我们

公开 SEO Tool 页面不能只说明“它怎么工作”，必须把产品价值讲出去。用户不关心内部流程本身，用户关心用了以后能得到什么、为什么这个结果值得信任、为什么不用别的工具或自己手动做。

每个 serious public Tool 页面必须有清晰的价值主张，并在页面主体中回答：

- 为什么用 Tomako 这个工具。
- 它比手工、模板、脚本、浏览器插件、普通文章或竞品更适合当前任务的地方。
- 什么样的同类工具才算好。
- Tomako 如何满足这些好工具标准。
- 用户最终得到的结果能帮他省什么、避免什么、提升什么。

“好工具标准”必须从用户真正关心的决策标准出发，而不是从产品内部流程出发。优先级通常是：

- **稳定性**：能不能稳定完成任务，失败率是否可控，失败时是否可恢复。
- **速度**：等待时间是否合理，流程是否少步骤。
- **结果质量**：文件/草案/报告/建议是否清晰、可用、可复核，质量是否尽量保真。
- **成本**：是否免费、低成本、少登录、少配置，是否减少人工或专业门槛。
- **易用性**：输入是否简单，结果是否能直接复制、下载、导出、继续编辑。
- **兼容性**：是否支持用户常见格式、平台、语言、地区、设备或工作流。
- **可信与安全**：隐私、版权、合规、数据处理、人工复核或平台边界是否清楚。

过程透明、状态说明、失败原因提示、边界解释是重要补充，但不能冒充主要标准。不要把“用户在输入前知道什么”“处理时知道发生了什么”这类过程信息写成最核心的好标准；它们应该服务稳定、速度、质量、易用或可信这些用户价值。

不同工具类型可以这样组织“好标准”：

- 下载/转换类：下载稳定、速度快、尽量保留原始质量、输出格式可用、免费或低成本、操作少、失败可恢复、版权/隐私边界清楚。
- 生成类：输出质量高、结构完整、贴合输入、可编辑、可导出、生成速度合理、缺失信息和风险可复核。
- 分析类：结论准确可用、来源清楚、严重程度有区分、建议可执行、能导出或交给下一步处理。
- 检查类：覆盖范围清楚、问题识别稳定、误报少、修复建议具体、支持复查、风险等级可信。

允许在需求不完整时做产品化假设和价值包装，但必须遵守：

- 可以把真实功能包装得更清楚、更有吸引力。
- 可以基于当前页面能力推导用户收益，例如少步骤、结构化结果、边界更清楚、失败提示更可读。
- 不得编造不可验证事实，例如官方合作、行业第一、最快、100% 成功、真实客户数据、第三方认证、法律保证、平台背书或具体性能数字。
- 不得承诺 runtime、后端、下载、抓取、生成质量或审核能力没有实现的东西。

“它是怎么工作的”不是必选模块。若保留，必须改写成用户结果导向，例如：

- “你会得到什么”
- “为什么这个结果更容易使用”
- “一个好的 X 工具应该具备什么”
- “使用前需要知道的边界”
- “下载/生成/分析失败时怎么办”

禁止把内部实现写给普通用户看，例如 FFmpeg、Agent 方案、LLM Task、Skills-OL、后端取舍、是否使用 Agent 等。除非目标用户明确是开发者，否则这些内容只属于开发记录。

可以从这些角度组织模块：

- 核心能力：工具真正完成了什么关键动作。
- 结果质量：输出为何可用、可复核、可导出、可继续修改。
- 差异化：比模板、手工、脚本、普通文章或竞品更好的地方。
- 成本收益：节省时间、减少重复劳动、降低试错、减少专业门槛。
- 专业感：结构化输出、风险提示、来源说明、格式支持、边界清楚。
- 工作流衔接：用户拿到结果后能复制、下载、发布、复核、继续编辑或进入下一个工具。

价值模块必须落到真实功能和输出，不要写成抽象品牌口号。

价值表达不能只出现在 FAQ 或页面底部。首屏 intro、工具工作台附近或主体前半部分至少要出现一次明确的“为什么值得用”。

## 用户疑问地图

页面必须主动猜想用户疑问，并在合适位置回答，不一定都放 FAQ。

常见疑问类型：

- 输入疑问：我需要提供什么？支持什么格式、链接、语言、地区？
- 输出疑问：会得到什么？能复制、下载、导出、继续编辑吗？
- 准确性疑问：结果可靠吗？什么时候会失败？失败后怎么办？
- 边界疑问：哪些内容不支持？哪些结果需要人工复核？
- 隐私疑问：提交的 URL、文本、文件会怎样处理？
- 成本疑问：是否免费、是否需要登录、是否要等待？
- 权限疑问：第三方平台、版权、商用、私密内容是否允许？
- 下一步疑问：拿到结果后应该怎么用，如何优化，如何进入后续流程？

疑问回答要放在用户需要它的位置：输入前回答“能不能用”，结果前回答“等什么”，结果后回答“怎么用”，页面底部再用 FAQ 收尾。

特别注意：失败排查和结果使用指导通常比普通 FAQ 更有 SEO 价值。不要只写“为什么失败？”这种一问一答，要让用户能照着页面完成下一步。

## 可抓取内容

重要 SEO 内容必须是 HTML 文本：

- H1。
- intro。
- 关键说明。
- 使用步骤。
- 输出解释。
- FAQ。
- 能力边界。
- 相关链接。

不要只把核心信息放在图片、canvas、客户端交互后隐藏状态或生成结果里。图片用于提高质感，不替代正文。

## Metadata

公开页面必须定义或继承：

- 本地化 title。
- 本地化 description。
- canonical。
- hreflang alternates。
- Open Graph。
- robots/index 决策。
- sitemap eligibility。
- 稳定且有价值时补 JSON-LD。

规则：

- title 包含具体任务和主要结果，不写空泛“免费工具”。
- description 说明输入、动作、输出和边界，像搜索结果里值得点击的一句话。
- metadata 不得承诺页面或运行时没有的能力。
- alt 描述图片有用主体，不堆关键词。
- `ToolSpec` 中的 locales 主要用于 SEO/list metadata，正文放 i18n message。

## FAQ

只有真实有搜索价值或用户决策价值时才加 FAQ。

FAQ 是收尾，不是主体解释区。输入格式、不支持范围、失败恢复、隐私、商用和复核要求应优先分散到工作台附近、“你会得到什么”、支持范围、结果使用指导或错误态中，FAQ 只补充用户最后仍可能搜索的问题。

常见好问题：

- 需要提供什么输入？
- 输出准确性如何？
- 能否复制、下载、商用？
- 提交的 URL 或数据会怎样处理？
- 这是 AI 生成还是确定性计算？
- 拿到结果后下一步做什么？
- 结果是否需要人工复核？
- 对第三方平台、下载、抓取或转换类工具：支持哪些链接/格式，为什么会失败，是否保存用户输入，是否能处理私密内容，用户使用下载内容需要注意什么。

每页至少 2 条 FAQ 必须是当前工具独有问题。例如问题里应出现具体输入、输出、平台、格式、场景或边界，而不是“这个工具免费吗？”这类所有工具都能复用的问题。

避免：

- 为了 SEO 强行写 FAQ。
- 问题重复关键词。
- 回答承诺未上线集成。
- 把 FAQ 写成长篇百科，埋掉工具本体。
- 用内部问题替代用户问题，例如“这个工具用到了 Agent 吗？”“为什么不用 LLM？”除非目标用户明确是开发者或隐私敏感用户。
- 把本应在主体中解释的核心价值全部塞进 FAQ。FAQ 是补充答疑，不是产品价值表达的主阵地。

## 多语言

本地化不是直译。

必须：

- 每个 locale 重新确认关键词和自然表达。
- 使用当地用户熟悉的产品分类、法规/地区词、例子和语气。
- H1、metadata、intro、FAQ、widget labels 在每个 locale 内一致。
- 避免保留英文习语、双关或内部缩写。
- 检查 `hreflang`、canonical、sitemap、Open Graph locale、URL 与文案匹配。

## 能力边界

当页面涉及 mock、demo、AI、抓取或专业领域，必须说明边界：

- 本地模板还是云端 Agent。
- 是否为草案。
- 是否需要人工/律师/财务/安全专业复核。
- URL 抓取可能受登录、robots、反爬或动态渲染影响。
- 输出不是保证、排名、法律意见、财务意见或安全审计结论。
- 第三方平台内容下载、解析或转换工具必须说明：只处理用户有权访问或使用的内容；私密、删除、受限、登录后内容可能失败；平台规则、版权和隐私责任由用户确认。

Agent-backed 工具的文案必须和真实 runtime 一致。脚本只能做确定性检查时，不要写成个性化专家建议。

能力一致性是底线：文案中每一句“能做什么”都必须在当前版本真实可用。能生成 Brief，就写“生成结构化 Brief”；不能直接生成高清成品图，就不能写“一键生成高清海报 PNG”。只能本地预览，就写“预览方案”，不能写“AI 已自动分析你的截图”。结果需要人工确认，就必须写“需要人工复核”，不能写“直接交付终稿”。

能力边界不能只藏在 FAQ。支持、不支持、隐私、商用、失败原因和人工复核要求，必须在用户做决定前出现在工作台附近、结果说明、支持范围或首屏摘要中。

## 内部链接

每个工具页应有：

- tools index 可达入口。
- related tools 或下一步 CTA。
- MCP guide link，当 MCP 启用且对用户有意义。
- 模板、博客、帮助文档链接，仅在帮助同一任务时加入。

页面集群按用户工作流互链，不按关键词机械互链。

相关工具应该服务用户下一步，而不是泛泛返回首页。例如下载类工具可链接到格式转换、压缩、素材整理；生成类工具可链接到改写、检查、导出；分析类工具可链接到修复、复查、报告生成。

## AI Search / GEO

把 GEO 当作 SEO 基本功：

- 顶部给出清晰答案和输出承诺。
- heading 层级清楚。
- section 短而自包含。
- 必要时使用例子、表格或列表帮助提取。
- 事实性、指导性内容优先引用权威来源。
- 不加隐藏文案或“LLM-friendly”废话块。
