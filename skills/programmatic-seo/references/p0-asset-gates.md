# P0 视觉物料门槛

## 使用时机

涉及 Hero 图、右侧 before/after 图、说明区配图、生图 prompt、截图、产品预览、视觉物料或图片验收时，必须读取本文件。

本文件吸收原设计师规范中的视觉和配图要求，并按最新 P0 调整：图片主体要撑满，不能带大背景底、外层卡片、宽留白或装饰 blob。生图物料以视觉表达为主，不是把 UI 截图画得更漂亮。

## 视觉物料的目标

SEO 工具页需要视觉物料来提升专业感和产品价值感，但图片不是信息说明书。

## 超级 P0：必须使用生图模型生成视觉物料

serious public Tool 页面的核心视觉物料必须使用生图模型生成 bitmap 图片。优先直接使用 Codex 原生生图能力生成，然后落地到 `public/tools/{slug}-{section}.png`。

禁止用前端代码、React 组件、CSS 卡片、SVG 流程图或 `ToolGuideVisual` 充当必需视觉物料。代码可以实现真实交互区，但不能替代页面配图。

如果当前环境暂时无法生图，必须把它记录为 blocker 或明确例外，不能静默改成代码画 UI。上线前仍要补齐真实生成图。

核心视觉图必须让用户感受到产品专业、结果可信、输出有价值，而不是重复说明流程。尤其禁止：

- 左侧文案已经说明功能，右侧又用“1、2、3、4”列表重写一遍。
- 把“输入 -> 处理 -> 输出”做成通用流程卡，但没有当前工具独有对象。
- 用技术架构、内部实现、Agent/LLM 方案图替代用户价值。
- 用密集 UI 小卡片假装高级，实际看不清结果。
- 把生图模型当成 UI 绘制器，只生成一张界面截图或一堆 icon。

更好的方向：

- 展示使用前后差异，例如零散链接/brief -> 清晰可下载文件、报告、草案、候选列表。
- 展示一个好结果应该具备的关键特征，例如格式清楚、状态明确、可复制/下载、风险可复核。
- 展示专业团队做产品的感觉，例如干净的结果预览、可信的边界提示、明确的下一步动作。
- 用对比、结果质量、支持范围或失败边界来补充正文，而不是复述正文。
- 用更场景化的表达，例如实物、人物、工作场景、材料变化、文件流动、前后对比、专业产物陈列或抽象但可感知的价值隐喻。

必须做到：

- 低信息密度。
- 高视觉中心。
- 能在 3 秒内感知工具价值。
- 图片主体足够大，放到页面里仍然清楚。
- 视觉表达为主，文字说明为辅；图片里的文字只保留极短标签。
- 不替代可抓取 HTML 文案。
- 表达当前工具的真实对象和结果，不要做成任何工具都能套用的通用流程图。
- 服务“产品价值”或“用户疑问”之一：要么让用户感到功能强、专业、结果可用，要么帮助用户更快理解怎么用、结果是什么、边界在哪里。

## 超级 P0：图片不是 UI 截图

生图模型的优势是视觉表达，不是复刻前端界面。除非明确需要展示真实产品截图，否则图片默认不要以完整 UI 面板、dashboard、表单或列表为主体。

优先选择：

- 场景化表达：人物正在完成任务、团队在整理材料、创作者保存素材、创业者准备上线文档等。
- 实物化表达：文件包、视频胶片、品牌卡片、报告夹、素材盒、清单板、下载文件、合规文件堆叠等。
- Before/after 对比：凌乱输入变成清晰结果，粗糙 brief 变成专业产物，分散素材变成可用文件。
- 价值隐喻：速度、清晰、可信、专业、少步骤、可复核、可交付。
- 少量产品感 UI 元素：只作为辅助，例如一个清晰结果卡、一个下载文件、一个简短标签。

避免：

- 纯 UI 截图。
- dashboard 拼贴。
- 密集卡片阵列。
- icon 堆叠。
- 只靠 UI 文字讲逻辑。
- 跟页面正文重复的信息图。

判断标准：去掉图片里的文字后，用户仍然能感受到“这个工具能带来什么价值”。如果去掉文字就看不懂，说明图片过度依赖信息表达，需要重做。

## 什么时候必须有图

serious public Tool 页面默认至少需要 4 张由生图模型生成的视觉物料，除非用户明确要求做极简内测页或该页面不面向公开 SEO。

4 张图的默认配置：

1. **Hero 背景图**：铺满 Hero 区或首屏大区域，作为 H1 / intro 的视觉背景。
2. **正文配图 A**：补充页面某段价值文案、结果说明或使用场景。
3. **正文配图 B**：补充支持范围、失败边界、结果质量或用户疑问。
4. **正文配图 C**：补充下一步、结果使用、对比、专业感或场景化价值。

前 3 张正文配图可以按页面内容自由选择位置，不需要每个 section 都有图，但必须服务具体段落，不能只是装饰。三张图之间要在构图、主体和表达目标上有差异。

第 4 张是 Hero 背景图，要求：

- 铺满 Hero 区大图，不是右侧小卡片。
- 页面 H1 / intro / CTA 可以作为 HTML 文本覆盖在图上。
- 图片本身也要有基础视觉元素，让用户一眼识别工具功能和价值。
- 视觉表达非常简洁，信息密度和文字密度都低。
- 能让用户感受到“这个页面是做什么的，我能从中得到什么”。
- 不要把 Hero 做成左右分栏的标题 + 卡片，不要用阅读时间、工具分类、更新时间、关键词标签占据 Hero 视觉。

优先需要图的位置：

- 复杂表单右侧 before/after 视觉。
- Hero 或首屏产品预览。
- How it works、Result guidance、Use cases 等说明区。
- 结果解读、判断标准、风险提示、输出示例。
- 产品价值模块，例如核心能力、差异化、输出质量、工作流衔接。
- 用户疑问模块，例如支持范围、格式、失败原因、隐私边界、复核清单。

不需要额外生成图的位置：

- 真实交互 widget 本身。
- 纯 icon、按钮、状态点。
- 已有真实产品截图且质量足够。

已有真实产品截图不能替代 Hero 背景图，除非它本身已经是高质量、低信息密度、可铺满 Hero 的视觉资产。

## 复杂表单右侧图

复杂表单的右侧图默认采用 before/after：

- Before：展示粗略输入或场景，例如 URL、产品名、短 brief、文件、关键词、选项、凌乱素材、待整理文档。
- After：展示输出类型或价值结果，例如清晰文件包、报告、候选列表、评分、文档预览、建议卡、可交付素材。
- 只用极短标签，不放段落。
- 不展示真实生成结果，真实结果在下方结果区。
- 主体贴近画布边缘，内容占据大部分图片。
- 右侧图不要只是把表单字段和输出字段画成 UI；它要表达“从混乱到清晰”“从输入到可用产物”的视觉差异。

## 生图前必须读取的信息

先从页面或 brief 提炼：

- H1。
- subtitle / intro。
- 核心关键词。
- 工具输入。
- 工具动作。
- 工具输出。
- use case。
- CTA 或结果价值。
- 该图片主要服务的目标：体现产品价值，还是解决用户疑问。
- 图片角色：Hero 背景图，还是正文配图 A/B/C。

每张图只表达 1 到 3 个核心元素：

- 输入。
- 输出。
- 价值感或场景。

超过 3 个信息点时必须删减。

提炼时必须使用当前工具的具体名词。例如下载器应出现“链接输入、媒体预览、格式选项、下载文件卡”；报告工具应出现“数据源、报告卡、风险/建议”；协议工具应出现“产品信息、文档草案、复核清单”。不要只写“input / AI / output”。

如果是价值型图片，优先展示结果质量、核心能力、前后差异、节省步骤或专业输出。
如果是疑问型图片，优先展示支持范围、失败/限制、风险提示、复核清单、隐私/权限边界或下一步指引。

## 图片文字语言

所有生图模型生成的图片，如果图片内部出现文字，必须使用英文。页面正文、H1、alt、i18n 文案仍按当前 locale 正常本地化；这里限制的是图片像素里的文字。

规则：

- 图片内文字优先少到没有；需要时只保留 1 到 3 个英文短标签。
- 不要在图片里放中文、日文、韩文或中英混排。
- 不要依赖图片文字承载 SEO 信息，SEO 信息必须在 HTML 文本中。
- 如果生图结果出现非英文文字、乱码、拼写错误或大段文字，必须重生或换无文字方案。

可用英文短标签示例：

- `Input`
- `Output`
- `Download`
- `Ready`
- `Draft`
- `Report`
- `Before`
- `After`
- `High Quality`
- `Fast Export`

## 构图原型

按页面上下文选择一个，不要每张都用同一结构。优先使用场景化或实物化构图，UI 构图只在确实有助于表达结果时使用：

- 场景化 before/after。
- 实物化文件包、素材包、报告包。
- 人物或团队使用场景。
- 品牌、内容、文档、媒体素材的视觉陈列。
- 输入 / 输出 before-after。
- 大结果卡。
- 报告或评分卡。
- 编辑器画布。
- 内容卡或社交卡。
- 时间线 / changelog。
- 轻流程三节点。
- 文档堆叠或文件包。

同一页面多张图必须变化构图、尺度或主视觉，不要连续重复中心节点连接图。

按工具类型选择更具体的视觉：

- 下载/转换/提取类：链接或文件输入 -> 媒体/文件预览 -> 格式选择或下载文件卡。
- 生成类：简短 brief -> 候选结果、文档草案或素材预览。
- 分析/报告类：数据源 -> 报告摘要、评分、风险或建议卡。
- 检查/清单类：待检查对象 -> 状态分组、问题等级、下一步清单。
- 编辑/优化类：before 内容 -> after 内容，二者必须能看出差异。

如果配图换掉工具名称后仍然成立，就太泛，需要重做。

## 画面约束

P0 禁止：

- 大面积装饰背景底。
- 外层大圆角卡片。
- 主体外再包一层浅色画布、展示台、背景板或网页容器。
- 图片里再套一层页面或截图容器。
- 宽内边距、厚白边、空画布。
- 彩色 blob、渐变球、bokeh、无关 scenery。
- 电脑、显示器、桌面、键盘、咖啡杯等物理设备场景，除非用户明确需要。
- 密集 dashboard、三列以上密集卡片、长表格。
- 大段文字、法律条文、政策全文、小到不可读的说明。
- Mac 窗口标题文字。
- 底部流程步骤。
- 水印。
- 星星、sparkle、starburst、彩屑，除非工具主题就是评分或收藏。
- 多个大面积近黑色卡片、按钮、流程节点或结果面板。
- 用通用步骤条、抽象节点、无意义卡片组冒充当前工具物料。
- 用一整张 UI 截图、dashboard、表单或列表作为主要画面。
- 只用 icon、按钮、标签和卡片表达价值。
- 在普通用户页面视觉中突出 Agent、LLM、内部评估、技术架构，而不是用户输入输出。

必须做到：

- 主体占据画布大部分面积。
- 主体几乎铺满图片安全区，不能为了“好看”留出大背景。
- 只有一个清晰视觉中心。
- 短标签可读，长文不出现。
- 图片和页面背景有区分，但不是靠额外背景底实现。
- 风格、圆角、阴影、色彩与页面一致。
- 如果有近黑填充，它只能是唯一焦点对象。
- 视觉中心能让用户相信工具真的能产出目标结果，例如可下载文件、媒体预览、报告卡、候选列表、文档草案。

## Prompt 结构

生成图片时使用中文理解、英文或中文 prompt 均可，但语义必须包含这些约束：

```text
用途：Tomako SEO 工具页面视觉物料，低信息密度，高视觉中心。
图片角色：[Hero background / inline visual A / inline visual B / inline visual C]
页面语境：[H1 / 输入 / 动作 / 输出]
图片目标：[体现产品价值 / 解决用户疑问]
价值重点：[为什么这个工具值得用 / 好结果长什么样 / 相比替代方案更清楚或更省事]
表达方式：[场景化 / 实物化 / 人物使用场景 / before-after / 少量产品 UI 辅助]
核心元素：只保留 1-3 个元素。
构图：[场景化 before-after / 实物文件包 / 人物工作场景 / 大结果物件 / 少量结果卡辅助等]
主体：内容几乎铺满画布安全区，贴近边缘，紧凑裁切，不要额外背景板。
文字：最多 1-3 个极短英文标签；不放中文、不放段落、不讲逻辑、不写步骤。
风格：高级、专业、有场景感，视觉表达强，结果可信。
禁止：大背景底、外层卡片、背景板、宽留白、blob、物理设备堆砌、长文、密集 dashboard、水印、星芒、多黑块、纯 UI 截图、icon 堆叠、1234 流程复读图、技术架构图。
```

Hero 背景图额外加入：

```text
Hero background: full-bleed wide composition for a website hero section.
Leave clean negative space for localized HTML H1 and intro overlay.
The image itself should still communicate the tool's function and user value at a glance.
No large text blocks, no UI dashboard, no card-in-card frame.
Any text inside the image must be English only.
```

## 文件落地

静态图片默认放在：

```text
public/tools/{slug}-{section}.png
```

推荐命名：

```text
public/tools/{slug}-hero.png
public/tools/{slug}-value.png
public/tools/{slug}-support.png
public/tools/{slug}-use-case.png
```

引用规则：

- 使用 `next/image` 或项目共享图片组件。
- 设置稳定宽高或 aspect-ratio，避免布局位移。
- `alt` 来自 i18n，描述图片的实际用途，不堆关键词。
- 不要再给图片外层加厚白底、内边距、大阴影或二次卡片。
- 图片文件本身不要自带额外背景底；页面容器负责排版，图片主体负责表达。
- 关键 SEO 内容仍在 HTML 文本中。

## 图片验收

交付前逐项检查：

- [ ] serious public Tool 页面至少有 4 张生图视觉物料：1 张 Hero 背景图 + 3 张正文配图。
- [ ] Hero 背景图铺满 Hero 区或首屏大区域，支持 H1 / intro 作为 HTML 文本覆盖。
- [ ] Hero 背景图本身能点题，能让用户识别功能和价值，不是抽象装饰。
- [ ] 3 张正文配图分别服务具体段落或模块，构图和表达目标不重复。
- [ ] 核心视觉物料来自生图模型生成的 bitmap 图片，不是前端代码、SVG、CSS 卡片或 `ToolGuideVisual`。
- [ ] 如果未能生图，已记录为 blocker 或上线前必须补齐的例外，没有静默用代码图替代。
- [ ] 图片内部如有文字，全部是英文短标签；没有中文、乱码、大段文字或拼写错误。
- [ ] 图片来自页面真实输入/输出，不是随机装饰。
- [ ] 图片明确服务产品价值或用户疑问，不只是填空装饰。
- [ ] 图片能让用户感受到专业感、结果质量或为什么值得使用。
- [ ] 图片以视觉表达为主，不靠文字、步骤或 UI 列表讲逻辑。
- [ ] 图片不是纯 UI 截图、dashboard、表单、icon 堆或代码画面。
- [ ] 图片有场景化、实物化、人物使用场景、before/after 或明确价值隐喻。
- [ ] 图片体现了当前工具对象，不能无损替换到另一类工具。
- [ ] 图片没有复读旁边正文，没有用“1、2、3、4”列表重复说明流程。
- [ ] 信息点不超过 3 个。
- [ ] 主体占据大部分图片，内容没有小小地漂在大背景里。
- [ ] 没有大背景底、背景板、外层卡片、blob、宽留白。
- [ ] 没有长文、密集表格、不可读小字。
- [ ] 没有电脑桌面等物理设备场景。
- [ ] 没有 Mac 标题、水印、底部步骤。
- [ ] 没有多个大黑块抢焦点。
- [ ] 没有把内部 Agent/LLM/技术评估做成主要视觉内容。
- [ ] 页面中图片尺寸稳定，移动端仍可读。
- [ ] 图片不替代可抓取 HTML 文案。
