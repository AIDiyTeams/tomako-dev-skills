# Tool 复盘模板

## 使用时机

处理 `docs/tool-retros/inbox`、用户转发项目复盘、页面多次失败、或需要更新 `programmatic-seo` Skill 时，必须读取本文件。

目标是抽象通用规则，不堆叠个案。

## 复盘输入建议

让项目执行者在阶段完成后提供：

```text
工具名称：
目标用户：
用户输入：
预期输出：
实际实现：
遇到的问题：
问题发生在哪一层：
最终解决方法：
仍未验证的风险：
涉及文件：
验证命令：
截图/日志/taskId/Skill Result：
希望沉淀成通用规则的点：
```

如果信息不完整，优先追问能帮助抽象通用规则的缺口：

- 是需求误判、UI 质量、运行时、Skill Result、部署、SEO 文案，还是上线流程问题？
- 这个问题是否会影响所有类似工具？
- 现有 Gate 是否已经覆盖，只是执行者没读，还是规则本身缺失？
- 需要改 Skill、reference、项目组件、模板、检查脚本，还是只需记录风险？

## 抽象规则方法

先分类，再更新：

- 需求层：是否需要更早确认输入输出、风险边界、运行时。
- 运行时层：是否误用前端模板、fallback、旧 task、raw output、异步状态。
- UI 层：是否表单太复杂、选项墙、选中态太重、组件库没用、结果区太挤。
- 物料层：是否图片信息太密、背景太大、主体太小、风格不对。
- 文案层：是否不可搜索、overpromise、内部术语、FAQ 空泛。
- 上线层：是否 slug、review/noindex、sitemap、deploy、cc-connect、Skills-OL 未验证。
- 流程层：是否因为未读文档、未做阅读状态、未做验收导致问题。

只把能预防同类问题的规则写入 Gate。不要把某个工具的具体实现、截图描述、一次性 bug、临时命令堆进通用文档。

## 更新判断

适合更新 Skill/Gate 的情况：

- 同类工具会反复遇到。
- 违反后会误导用户、导致不可用、无法上线、SEO 低质或结果不可信。
- 现有文档没有覆盖。
- 现有文档覆盖了但位置太深、表达不够硬、读取路径不清。

不适合更新 Skill/Gate 的情况：

- 只属于某个组件的一次性 bug。
- 项目代码需要修而不是文档规则缺失。
- 已有 P0 明确覆盖，执行者只是没读；这时应加强阅读状态或验收，而不是重复写规则。
- 个案命令、路径、日志不具备复用价值。

## 输出格式

复盘完成后输出：

```text
通用问题：
- ...

已更新：
- 文件：
- 新规则：

未写入通用规则：
- 原因：

仍建议项目侧修复：
- ...

验证：
- ...
```

## P0 升级标准

把规则升为 P0 的条件：

- 会让用户得不到真实结果。
- 会让页面误导用户或搜索引擎。
- 会让高风险内容被错误处理。
- 会让工具无法上线或被错误索引。
- 会让页面明显难用、不敢用、无法读结果。
- 已在多个项目或同一项目多次复现。

如果只是风格偏好但不影响可用性、可信度、上线或搜索价值，放入 P1 或具体设计参考，不要污染 P0。
